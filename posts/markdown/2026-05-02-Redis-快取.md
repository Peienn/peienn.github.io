# Redis 快取雪崩、穿透與擊穿：三大災難的防禦實戰指南

## 前言

在高流量的生產環境中，Redis 快取是保護後端資料庫的關鍵防線。然而，當這道防線失效時，瞬間湧入的請求可能直接壓垮資料庫，導致整個服務癱瘓。身為 SRE，我曾在凌晨三點被 PagerDuty 叫醒，原因正是 Redis 快取大量同時過期，造成資料庫 CPU 飆升到 100%。

快取雪崩（Cache Avalanche）、快取穿透（Cache Penetration）、快取擊穿（Cache Breakdown）是三種最常見且最具破壞力的快取失效場景。理解這三者的差異並建立對應的防禦機制，是每位中階 DevOps 工程師必備的技能。

## 快取雪崩：大量 Key 同時過期的災難

快取雪崩發生在大量快取 Key 在同一時間點過期，導致所有請求直接打到資料庫。最常見的情境是系統重啟後批次載入快取，或是定時任務統一設定相同的 TTL。

**解決方案：為 TTL 加入隨機偏移量**

```python
import random

def set_cache_with_jitter(redis_client, key, value, base_ttl=3600):
    # 在基礎 TTL 上增加 0~300 秒的隨機偏移
    jitter = random.randint(0, 300)
    actual_ttl = base_ttl + jitter
    redis_client.setex(key, actual_ttl, value)
```

另一個有效策略是實作多層快取架構，搭配本地快取（如 Caffeine）作為第一層防線：

```yaml
# 應用層快取配置範例
cache:
  local:
    type: caffeine
    ttl: 60s
    max-size: 10000
  remote:
    type: redis
    ttl: 3600s
```

## 快取穿透：查詢不存在資料的攻擊向量

快取穿透是指查詢一個資料庫中根本不存在的資料。由於資料不存在，自然無法被快取，每次請求都會穿透到資料庫。惡意攻擊者可以利用這點，用大量不存在的 Key 發動攻擊。

**解決方案一：快取空值**

```python
def get_user(redis_client, db, user_id):
    cache_key = f"user:{user_id}"
    cached = redis_client.get(cache_key)
    
    if cached is not None:
        if cached == "NULL":
            return None
        return json.loads(cached)
    
    user = db.query_user(user_id)
    if user is None:
        # 快取空值，但設定較短的 TTL
        redis_client.setex(cache_key, 300, "NULL")
        return None
    
    redis_client.setex(cache_key, 3600, json.dumps(user))
    return user
```

**解決方案二：布隆過濾器前置檢查**

```bash
# 使用 RedisBloom 模組
redis-cli BF.ADD user_filter user:1001
redis-cli BF.ADD user_filter user:1002

# 查詢前先檢查
redis-cli BF.EXISTS user_filter user:9999
# 返回 0 表示絕對不存在，直接拒絕請求
```

## 快取擊穿：熱點 Key 失效的瞬間風暴

快取擊穿專指某個熱點 Key（如熱門商品、首頁資料）在過期的瞬間，大量併發請求同時發現快取失效，全部湧向資料庫重建快取。

**解決方案：分散式鎖 + 單一重建**

```python
import redis
import time

def get_hot_data_with_lock(redis_client, db, key):
    data = redis_client.get(key)
    if data:
        return json.loads(data)
    
    lock_key = f"lock:{key}"
    # 嘗試獲取分散式鎖，設定 10 秒自動釋放
    if redis_client.set(lock_key, "1", nx=True, ex=10):
        try:
            # 雙重檢查
            data = redis_client.get(key)
            if data:
                return json.loads(data)
            
            # 從資料庫載入並重建快取
            data = db.query_hot_data()
            redis_client.setex(key, 3600, json.dumps(data))
            return data
        finally:
            redis_client.delete(lock_key)
    else:
        # 未獲得鎖，等待後重試
        time.sleep(0.1)
        return get_hot_data_with_lock(redis_client, db, key)
```

對於極度熱門的資料，也可以考慮設定永不過期，透過背景任務主動更新：

```python
# 背景更新任務
def refresh_hot_cache(redis_client, db):
    while True:
        data = db.query_hot_data()
        redis_client.set("hot:data", json.dumps(data))
        time.sleep(60)  # 每分鐘主動更新
```

## 監控與告警：建立早期預警機制

防禦機制需要搭配完善的監控才能發揮作用。以下是關鍵的監控指標：

```bash
# 監控快取命中率
redis-cli INFO stats | grep keyspace_hits
redis-cli INFO stats | grep keyspace_misses

# 計算命中率
# hit_rate = keyspace_hits / (keyspace_hits + keyspace_misses)
```

建議在 Prometheus + Grafana 中設定告警規則：

```yaml
# prometheus alert rules
groups:
  - name: redis_cache_alerts
    rules:
      - alert: CacheHitRateLow
        expr: redis_keyspace_hits_total / (redis_keyspace_hits_total + redis_keyspace_misses_total) < 0.8
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Redis 快取命中率低於 80%"
```

## 結論與行動建議

快取雪崩、穿透、擊穿這三種問題的本質都是「請求未能被快取攔截而直接衝擊資料庫」。針對不同場景，我們需要採取不同策略：

1. **雪崩**：TTL 加入隨機偏移、多層快取架構
2. **穿透**：快取空值、布隆過濾器前置過濾
3. **擊穿**：分散式鎖控制重建、熱點資料永不過期

建議立即檢視你的生產環境，確認是否已經實作這些防禦機制。同時，定期進行混沌工程演練，模擬大量快取失效的場景，驗證系統的韌性
