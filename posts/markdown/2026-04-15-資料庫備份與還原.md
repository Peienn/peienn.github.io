# PostgreSQL 時間點還原（PITR）實戰：拯救你的誤刪資料

## 前言

「糟糕，我剛才執行的 DELETE 忘記加 WHERE 條件了...」這句話大概是每位 DBA 最不想聽到的惡夢。傳統的每日完整備份雖然能還原到昨天的狀態，但這意味著今天所有的交易紀錄都將付諸流水。

這時候，**時間點還原（Point-in-Time Recovery，PITR）** 就是你的救命稻草。透過 PostgreSQL 的 WAL（Write-Ahead Logging）機制，你可以將資料庫還原到任何指定的時間點——精確到那筆誤刪指令執行前的那一秒。本文將帶你從零開始設定 PITR，並實際演練一次完整的還原流程。

## 理解 PITR 的運作原理

PostgreSQL 的 PITR 建立在兩個關鍵元素之上：**基礎備份（Base Backup）** 與 **WAL 日誌歸檔**。

基礎備份是資料庫在某個時間點的完整快照，而 WAL 則記錄了所有資料變更的操作日誌。還原時，PostgreSQL 會先載入基礎備份，然後依序重播 WAL 日誌，直到你指定的時間點為止。

```
[基礎備份] → [WAL.001] → [WAL.002] → ... → [目標時間點] → 停止重播
   ↑                                              ↑
  週日凌晨                                    週三下午 2:34:56
```

這套機制讓你能夠在數日甚至數週後，還原到任意精確時間點的資料狀態。

## 設定 WAL 歸檔機制

首先，我們需要啟用 WAL 歸檔功能。編輯 `postgresql.conf`：

```bash
# 啟用歸檔模式
wal_level = replica
archive_mode = on
archive_command = 'cp %p /var/lib/postgresql/wal_archive/%f'

# 建議同時設定，確保 WAL 不會累積過多
max_wal_size = 2GB
min_wal_size = 1GB
```

建立歸檔目錄並設定權限：

```bash
sudo mkdir -p /var/lib/postgresql/wal_archive
sudo chown postgres:postgres /var/lib/postgresql/wal_archive
sudo systemctl restart postgresql
```

驗證歸檔是否正常運作：

```sql
-- 強制切換 WAL 段落
SELECT pg_switch_wal();

-- 確認歸檔狀態
SELECT * FROM pg_stat_archiver;
```

## 建立基礎備份

有了持續的 WAL 歸檔後，接下來需要建立基礎備份。使用 `pg_basebackup` 是最簡單的方式：

```bash
pg_basebackup -D /var/lib/postgresql/backup/base_20260415 \
  -Ft -z \
  -P \
  -X stream \
  -c fast \
  -l "weekly_backup_20260415"
```

參數說明：
- `-Ft -z`：以 tar 格式輸出並啟用 gzip 壓縮
- `-X stream`：同時串流備份期間產生的 WAL
- `-c fast`：快速執行 checkpoint，減少備份時間
- `-P`：顯示備份進度

建議透過 cron 每週執行一次基礎備份：

```bash
0 3 * * 0 pg_basebackup -D /backup/base_$(date +\%Y\%m\%d) -Ft -z -X stream -c fast
```

## 執行時間點還原

假設災難發生了——有人在下午 14:30 執行了誤刪操作，而你需要還原到 14:29:30 的狀態。

**步驟一：停止 PostgreSQL 並備份現有資料**

```bash
sudo systemctl stop postgresql
mv /var/lib/postgresql/16/main /var/lib/postgresql/16/main_corrupted
```

**步驟二：還原基礎備份**

```bash
mkdir /var/lib/postgresql/16/main
tar -xzf /backup/base_20260413/base.tar.gz -C /var/lib/postgresql/16/main
tar -xzf /backup/base_20260413/pg_wal.tar.gz -C /var/lib/postgresql/16/main/pg_wal
```

**步驟三：設定還原參數**

建立 `recovery.signal` 並編輯 `postgresql.conf`：

```bash
touch /var/lib/postgresql/16/main/recovery.signal
```

```bash
# 加入 postgresql.conf
restore_command = 'cp /var/lib/postgresql/wal_archive/%f %p'
recovery_target_time = '2026-04-15 14:29:30+08'
recovery_target_action = 'promote'
```

**步驟四：啟動並完成還原**

```bash
chown -R postgres:postgres /var/lib/postgresql/16/main
sudo systemctl start postgresql
```

查看日誌確認還原完成：

```bash
tail -f /var/log/postgresql/postgresql-16-main.log
# 應該看到：database system is ready to accept connections
```

## 結論與行動建議

PITR 是 PostgreSQL 生產環境中不可或缺的保護機制，但它需要事先設定才能在災難發生時派上用場。以下是你的行動清單：

1. **今天就啟用 WAL 歸檔**——這是 PITR 的基礎，沒有歸檔就沒有還原的可能
2. **建立基礎備份排程**——建議每週一次，搭配每日的邏輯備份（pg_dump）形成多層防護
3. **定期演練還原流程**——在測試環境每季執行一次完整的 PITR 演練，確保團隊熟悉操作步驟
4. **監控歸檔空間**——WAL 歸檔會持續增長，務必設定磁碟空間警報並定期清理過期檔案

記住：備份的價值不在於備份本身，而在於你能夠成功還原的那一刻。
