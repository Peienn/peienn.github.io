# MySQL 慢查詢排查實戰：從發現到優化的完整指南

## 前言

凌晨三點，監控告警響起，資料庫 CPU 飆到 95%，API 回應時間從 50ms 暴漲到 5 秒。身為 SRE，你第一時間登入主機，卻發現數十條 SQL 同時在跑，該從何下手？

慢查詢（Slow Query）是 MySQL 最常見的效能殺手。它不像硬體故障那樣明顯，往往在系統負載升高時才浮現，但一旦發生，就會像骨牌效應般拖垮整個服務。本文將帶你建立一套系統化的慢查詢排查流程，從即時診斷到長期監控，讓你下次遇到類似情況時能快速定位並解決問題。

## 第一步：啟用並分析慢查詢日誌

MySQL 內建的 Slow Query Log 是排查的第一道防線。許多生產環境預設關閉此功能，建議在可接受的效能損耗下啟用它。

```sql
-- 檢查當前設定
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 動態啟用（無需重啟）
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 記錄超過 1 秒的查詢
SET GLOBAL log_queries_not_using_indexes = 'ON';
```

日誌啟用後，使用 `mysqldumpslow` 快速分析：

```bash
# 找出執行次數最多的前 10 條慢查詢
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

# 找出平均執行時間最長的查詢
mysqldumpslow -s at -t 10 /var/log/mysql/slow.log
```

這能讓你在大量日誌中快速聚焦真正的問題查詢，而非被單一偶發的慢查詢誤導。

## 第二步：使用 EXPLAIN 解析執行計畫

定位到可疑的 SQL 後，用 `EXPLAIN` 分析其執行計畫是關鍵步驟。

```sql
EXPLAIN SELECT * FROM orders 
WHERE customer_id = 12345 
AND created_at > '2026-01-01'
ORDER BY total_amount DESC;
```

重點關注以下欄位：

| 欄位 | 警示信號 |
|------|----------|
| type | 出現 `ALL` 表示全表掃描 |
| rows | 數值過大代表掃描過多資料 |
| Extra | `Using filesort`、`Using temporary` 需注意 |
| key | `NULL` 表示未使用任何索引 |

進階技巧是使用 `EXPLAIN ANALYZE`（MySQL 8.0.18+），它會實際執行查詢並顯示真實的時間消耗：

```sql
EXPLAIN ANALYZE SELECT * FROM orders 
WHERE customer_id = 12345;
```

## 第三步：即時監控當前執行中的查詢

當系統正在卡頓時，你需要看的是「現在」正在發生什麼，而非歷史日誌。

```sql
-- 查看當前所有連線與執行中的查詢
SHOW FULL PROCESSLIST;

-- 更精確的方式（MySQL 5.7+）
SELECT * FROM information_schema.PROCESSLIST 
WHERE COMMAND != 'Sleep' 
AND TIME > 5  -- 執行超過 5 秒
ORDER BY TIME DESC;

-- 如需終止問題查詢
KILL QUERY <process_id>;
```

搭配 Performance Schema 可獲得更詳細的資訊：

```sql
-- 找出消耗資源最多的 SQL 語句
SELECT DIGEST_TEXT, COUNT_STAR, AVG_TIMER_WAIT/1000000000 AS avg_ms
FROM performance_schema.events_statements_summary_by_digest
ORDER BY AVG_TIMER_WAIT DESC
LIMIT 10;
```

## 第四步：建立有效的索引策略

90% 的慢查詢問題可透過正確的索引解決。根據分析結果，建立複合索引時遵循「最左前綴原則」：

```sql
-- 針對前面的查詢範例，建立複合索引
CREATE INDEX idx_customer_created ON orders(customer_id, created_at);

-- 確認索引使用狀況
SELECT * FROM sys.schema_unused_indexes;
SELECT * FROM sys.schema_redundant_indexes;
```

常見的索引陷阱：
- 對低基數欄位（如 `status`、`gender`）單獨建索引效果有限
- `LIKE '%keyword%'` 無法使用索引，考慮全文索引或 Elasticsearch
- 隱式型別轉換會導致索引失效（字串欄位用數字查詢）

## 第五步：整合監控與長期追蹤

單次排查解決的是眼前問題，建立長期監控才能防患未然。推薦使用 Percona Monitoring and Management（PMM）或整合 Prometheus + Grafana：

```yaml
# prometheus.yml 設定範例
scrape_configs:
  - job_name: 'mysql'
    static_configs:
      - targets: ['mysql-exporter:9104']
```

關鍵監控指標：
- `mysql_global_status_slow_queries`：慢查詢累計次數
- `mysql_global_status_questions`：每秒查詢數（QPS）
- `mysql_global_status_threads_running`：活躍執行緒數

## 結論與行動建議

慢查詢排查是一項需要系統化方法的工作。建議你現在就採取以下行動：

1. **今天**：檢查生產環境的 `slow_query_log` 是否啟用
2. **本週**：部署 PMM 或設定 Grafana 儀表板監控關鍵指標
3. **持續**：將慢查詢分析納入每週例行檢查，主動發現潛在問題

記住，最好的故障處理是讓故障不發生。透過主動監控與定期優化，你可以把凌晨三點的告警消滅在萌芽階段。
