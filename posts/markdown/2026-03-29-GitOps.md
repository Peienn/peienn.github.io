# GitOps 環境中的 Secret 管理困境：Sealed Secrets vs External Secrets Operator 實戰比較

## 前言

當團隊開始導入 GitOps 流程時，往往會遇到一個棘手的問題：「機敏資料該怎麼存進 Git Repository？」

GitOps 的核心理念是將所有基礎設施與應用配置都納入版本控制，讓 Git 成為唯一的 Single Source of Truth。然而，資料庫密碼、API 金鑰、TLS 憑證等機敏資料，顯然不能以明文形式 commit 進 Repository。這個看似簡單的問題，卻是許多團隊在 GitOps 旅程中的第一道關卡。

本文將深入比較兩個主流解決方案：Bitnami Sealed Secrets 與 External Secrets Operator (ESO)，幫助你根據團隊情境做出適合的選擇。

## Sealed Secrets：將加密後的 Secret 直接存入 Git

Sealed Secrets 由 Bitnami 開發，核心概念是在 Cluster 內運行一個 Controller，只有該 Controller 持有私鑰能夠解密。開發者使用 `kubeseal` CLI 將原始 Secret 加密成 SealedSecret 資源，這個加密後的 YAML 可以安全地 commit 進 Git。

安裝方式相當直覺：

```bash
# 安裝 Sealed Secrets Controller
helm repo add sealed-secrets https://bitnami-labs.github.io/sealed-secrets
helm install sealed-secrets sealed-secrets/sealed-secrets \
  --namespace kube-system

# 安裝 kubeseal CLI (macOS)
brew install kubeseal
```

實際使用時，你會先建立一般的 Secret YAML，再用 kubeseal 轉換：

```bash
# 建立原始 Secret（不要 commit 這個檔案！）
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password=supersecret \
  --dry-run=client -o yaml > secret.yaml

# 加密成 SealedSecret
kubeseal --format yaml < secret.yaml > sealed-secret.yaml

# 現在可以安全地 commit sealed-secret.yaml
git add sealed-secret.yaml && git commit -m "Add encrypted db credentials"
```

## External Secrets Operator：從外部 Secret Manager 同步

ESO 採取不同策略：它不在 Git 中存放加密後的 Secret 內容，而是存放「從哪裡取得 Secret」的參照資訊。實際的機敏資料存放在 AWS Secrets Manager、HashiCorp Vault、GCP Secret Manager 等外部服務中。

```bash
# 安裝 External Secrets Operator
helm repo add external-secrets https://charts.external-secrets.io
helm install external-secrets external-secrets/external-secrets \
  --namespace external-secrets --create-namespace
```

首先需要設定 SecretStore，告訴 ESO 如何連接你的 Secret Manager：

```yaml
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: aws-secrets-manager
spec:
  provider:
    aws:
      service: SecretsManager
      region: ap-northeast-1
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets-sa
```

接著建立 ExternalSecret，定義要同步哪些 Secret：

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: SecretStore
  target:
    name: db-credentials
  data:
    - secretKey: username
      remoteRef:
        key: production/database
        property: username
    - secretKey: password
      remoteRef:
        key: production/database
        property: password
```

## 兩者的關鍵差異與適用場景

| 面向 | Sealed Secrets | External Secrets Operator |
|------|----------------|---------------------------|
| Secret 儲存位置 | Git Repository（加密） | 外部 Secret Manager |
| 金鑰管理 | Controller 自動產生，需備份 | 依賴外部服務的 IAM |
| Secret 輪換 | 需重新 seal 並 commit | 自動同步，支援 refreshInterval |
| 多叢集部署 | 每個 Cluster 金鑰不同，需個別 seal | 多叢集可共用同一 Secret Store |
| 學習曲線 | 較低，獨立運作 | 較高，需整合外部服務 |

**選擇 Sealed Secrets 的情境**：小型團隊、單一叢集、沒有現成 Secret Manager、希望 Secret 完全自包含在 Git 中。

**選擇 ESO 的情境**：企業環境、多叢集架構、已有 Vault 或雲端 Secret Manager、需要自動輪換 Secret、跨團隊共享 Secret。

## 混合使用的進階模式

實務上，兩者並非互斥。許多團隊採用混合策略：

- 開發/測試環境使用 Sealed Secrets，減少對外部服務的依賴
- 正式環境使用 ESO 連接 HashiCorp Vault，滿足稽核與自動輪換需求

你可以在 Kustomize 中透過 overlay 輕鬆切換：

```yaml
# base/kustomization.yaml
resources:
  - deployment.yaml

# overlays/dev/kustomization.yaml  
resources:
  - ../../base
  - sealed-secret.yaml

# overlays/prod/kustomization.yaml
resources:
  - ../../base
  - external-secret.yaml
```

## 結論與行動建議

Secret 管理是 GitOps 實踐中不可迴避的議題。Sealed Secrets 提供了最低門檻的入門方案，讓你快速在 Git 中安全存放機敏資料；External Secrets Operator 則提供了企業級的彈性，與現有的 Secret 管理基礎設施無縫整合。

**建議的下一步行動**：

1. 如果團隊尚未有任何 Secret 管理方案，先從 Sealed Secrets 開始，兩小時內即可上手
2. 如果公司已有 Vault 或雲端 Secret Manager，直接導入 ESO 能避免重複建設
3. 無論選擇哪種方案，務必建立 Secret 的備份與災難復原流程，尤其是 Sealed Secrets 的 Controller 私鑰

GitOps 的價值在於一致性與可追溯性，選擇正確的 Secret 管理策略，才能讓這份價值延伸到整個基礎設施。
