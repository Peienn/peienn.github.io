# gRPC 服務健康檢查實戰：從零建置完整的 Health Check 機制

## 前言

在微服務架構中，服務的健康檢查是維運的基石。然而，當你從 REST API 轉向 gRPC 時，傳統的 HTTP 健康檢查端點突然變得不適用。Kubernetes 的 `httpGet` probe 無法直接與 gRPC 服務溝通，這讓許多團隊在第一次部署 gRPC 服務時踩到坑。

本文將深入探討如何為 gRPC 服務建置標準化的健康檢查機制，涵蓋 gRPC Health Checking Protocol、Kubernetes 整合，以及實際的除錯技巧。

## 理解 gRPC Health Checking Protocol

gRPC 官方定義了標準的健康檢查協議（[GRPC Health Checking Protocol](https://github.com/grpc/grpc/blob/master/doc/health-checking.md)），這是一個內建的服務定義，讓客戶端可以查詢服務狀態。

核心概念是實作 `grpc.health.v1.Health` 服務，它提供兩個方法：

```protobuf
service Health {
  rpc Check(HealthCheckRequest) returns (HealthCheckResponse);
  rpc Watch(HealthCheckRequest) returns (stream HealthCheckResponse);
}

message HealthCheckResponse {
  enum ServingStatus {
    UNKNOWN = 0;
    SERVING = 1;
    NOT_SERVING = 2;
    SERVICE_UNKNOWN = 3;
  }
  ServingStatus status = 1;
}
```

以 Go 語言為例，實作健康檢查服務相當直觀：

```go
import (
    "google.golang.org/grpc/health"
    "google.golang.org/grpc/health/grpc_health_v1"
)

func main() {
    server := grpc.NewServer()
    
    // 建立健康檢查服務
    healthServer := health.NewServer()
    grpc_health_v1.RegisterHealthServer(server, healthServer)
    
    // 設定服務狀態
    healthServer.SetServingStatus("my-service", grpc_health_v1.HealthCheckResponse_SERVING)
    
    // 當依賴服務不可用時，更新狀態
    healthServer.SetServingStatus("my-service", grpc_health_v1.HealthCheckResponse_NOT_SERVING)
}
```

## Kubernetes 原生 gRPC Probe 設定

從 Kubernetes 1.24 版本開始，gRPC 健康檢查已成為正式功能。你可以直接在 Pod spec 中使用 `grpc` 類型的 probe：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: grpc-service
spec:
  containers:
  - name: app
    image: my-grpc-service:latest
    ports:
    - containerPort: 50051
    livenessProbe:
      grpc:
        port: 50051
        service: "my-service"  # 可選，空字串檢查整體服務
      initialDelaySeconds: 10
      periodSeconds: 10
    readinessProbe:
      grpc:
        port: 50051
        service: "my-service"
      initialDelaySeconds: 5
      periodSeconds: 5
```

需要注意的是，如果你的 gRPC 服務啟用了 TLS，Kubernetes 1.27+ 支援透過 annotation 指定：

```yaml
metadata:
  annotations:
    "grpc.health.check.tls": "true"
```

## 使用 grpcurl 進行手動診斷

當健康檢查失敗時，`grpcurl` 是最實用的除錯工具。它類似於 `curl`，但專門用於 gRPC 服務：

```bash
# 安裝 grpcurl
brew install grpcurl  # macOS
# 或
go install github.com/fullstorydev/grpcurl/cmd/grpcurl@latest

# 檢查服務健康狀態
grpcurl -plaintext localhost:50051 grpc.health.v1.Health/Check

# 檢查特定服務
grpcurl -plaintext -d '{"service": "my-service"}' \
  localhost:50051 grpc.health.v1.Health/Check

# 列出所有可用服務（需啟用 reflection）
grpcurl -plaintext localhost:50051 list
```

輸出範例：

```json
{
  "status": "SERVING"
}
```

若服務未啟用 gRPC reflection，你需要提供 proto 文件：

```bash
grpcurl -plaintext -proto health.proto \
  localhost:50051 grpc.health.v1.Health/Check
```

## 整合依賴服務的健康狀態

實務上，服務的健康狀態往往取決於下游依賴。以下是一個整合資料庫連線檢查的範例：

```go
func (s *Server) startHealthChecker(ctx context.Context, healthServer *health.Server) {
    ticker := time.NewTicker(5 * time.Second)
    defer ticker.Stop()
    
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            status := grpc_health_v1.HealthCheckResponse_SERVING
            
            // 檢查資料庫連線
            if err := s.db.PingContext(ctx); err != nil {
                log.Printf("Database health check failed: %v", err)
                status = grpc_health_v1.HealthCheckResponse_NOT_SERVING
            }
            
            // 檢查 Redis 連線
            if err := s.redis.Ping(ctx).Err(); err != nil {
                log.Printf("Redis health check failed: %v", err)
                status = grpc_health_v1.HealthCheckResponse_NOT_SERVING
            }
            
            healthServer.SetServingStatus("my-service", status)
        }
    }
}
```

## 結論與行動建議

為 gRPC 服務建置健康檢查機制是生產環境部署的必要條件。以下是建議的行動步驟：

1. **立即實作**：在每個 gRPC 服務中加入標準的 Health Checking Protocol
2. **升級 Kubernetes**：確保叢集版本 >= 1.24 以使用原生 gRPC probe
3. **建立監控**：將健康檢查狀態整合到 Prometheus metrics 中
4. **文件化**：記錄每個服務的健康檢查邏輯，包含哪些依賴會影響狀態

健康檢查看似簡單，卻是服務可靠性的關鍵。投資時間建置完善的健康檢查機制，將大幅降低生產環境的故障排除時間。
