# GitOps 實戰：如何優雅處理 Secrets 管理的三種方案比較

## 前言

當團隊開始導入 GitOps 工作流程後，很快就會遇到一個棘手的問題：「敏感資料該怎麼存？」

GitOps 的核心理念是將所有配置都存放在 Git 儲存庫中，作為 Single Source of Truth。但這個理念碰上 database 密碼、API keys、TLS 憑證等敏感資訊時，就產生了矛盾——我們不可能把明文密碼直接 commit 到 Git 裡。

這個問題若處理不當，輕則造成開發流程斷裂（部分配置在 Git、部分在別處），重則導致嚴重的資安事件。本文將比較三種主流的 GitOps Secrets 管理方案，幫助你根據團隊規模與需求做出合適的選擇。

## 方案一：Sealed Secrets — 輕量入門首選

Bitnami 開發的 Sealed Secrets 是最容易上手的方案。它的原理是在 cluster 內運行一個 controller，持有私鑰；開發者使用對應的公鑰加密 secrets 後，就可以安全地將加密後的 `SealedSecret` 物件存入 Git。

安裝方式相當簡單：

```bash
# 安裝 controller 到 cluster
helm repo add sealed-secrets https://bitnami-labs.github.io/sealed-secrets
helm install sealed-secrets sealed-secrets/sealed-secrets -n kube-system

# 本機安裝 kubeseal CLI
brew install kubeseal
```

加密 secret 的流程如下：

```bash
# 先建立一般的 secret yaml（不要 commit 這個檔案！）
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password=s3cr3t \
  --dry-run=client -o yaml > db-creds.yaml

# 使用 kubeseal 加密
kubeseal --format yaml < db-creds.yaml > db-creds-sealed.yaml

# 現在可以安全地 commit db-creds-sealed.yaml
```

**優點**：學習曲線低、不需要額外基礎設施。  
**缺點**：secrets 與特定 cluster 綁定，多叢集環境下管理複雜。

## 方案二：External Secrets Operator — 企業級整合

如果你的組織已經在使用 AWS Secrets Manager、HashiCorp Vault 或 GCP Secret Manager，External Secrets Operator（ESO）會是更好的選擇。它讓 Kubernetes 能夠直接從外部 secrets 儲存服務同步敏感資料。

```bash
# 安裝 ESO
helm repo add external-secrets https://charts.external-secrets.io
helm install external-secrets external-secrets/external-secrets -n external-secrets --create-namespace
```

以整合 AWS Secrets Manager 為例，首先建立連線配置：

```yaml
# secretstore.yaml
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: aws-secrets-manager
spec:
  provider:
    aws:
      service: SecretsManager
      region: ap-northeast-1
      auth:
        jwt:
          serviceAccountRef:
            name: external-secrets-sa
```

接著定義要同步哪些 secrets：

```yaml
# external-secret.yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: SecretStore
  target:
    name: db-creds
  data:
    - secretKey: password
      remoteRef:
        key: production/database
        property: password
```

**優點**：集中管理、支援 secrets rotation、完整 audit log。  
**缺點**：需要維護外部服務、架構複雜度較高。

## 方案三：SOPS + Age — 開發者友善的加密方案

Mozilla SOPS 配合 age 加密工具，提供了一個優雅的檔案層級加密方案。它只加密 YAML/JSON 中的 value，保留 key 為明文，讓 code review 時仍能理解檔案結構。

```bash
# 安裝工具
brew install sops age

# 產生加密金鑰對
age-keygen -o keys.txt
# 輸出類似：# public key: age1xxxxxxx
```

建立 `.sops.yaml` 設定檔指定加密規則：

```yaml
# .sops.yaml
creation_rules:
  - path_regex: .*secrets.*\.yaml$
    age: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
```

日常使用：

```bash
# 編輯加密檔案（自動解密、儲存時加密）
sops secrets/db-creds.yaml

# 搭配 ArgoCD 使用時，安裝 KSOPS plugin
# 讓 ArgoCD 能在部署時自動解密
```

**優點**：Git diff 友善、支援多種加密後端（age、PGP、AWS KMS）。  
**缺點**：需要管理加密金鑰的發佈與 rotation。

## 方案比較與選擇建議

| 面向 | Sealed Secrets | External Secrets | SOPS + Age |
|------|----------------|------------------|------------|
| 學習成本 | 低 | 中 | 中 |
| 多叢集支援 | 差 | 優 | 良 |
| 外部依賴 | 無 | 需要 Vault/雲端服務 | 無 |
| Secrets Rotation | 手動 | 自動 | 手動 |
| 適合規模 | 小型團隊 | 中大型企業 | 中小型團隊 |

## 結論與行動建議

沒有完美的方案，只有最適合你情境的選擇。我的建議是：

1. **剛開始導入 GitOps 的小團隊**：從 Sealed Secrets 開始，快速建立流程
2. **已有 Vault 或雲端 Secrets 服務的組織**：採用 External Secrets Operator 整合既有基礎設施
3. **重視開發體驗且願意管理金鑰的團隊**：SOPS + Age 提供最佳的 Git 工作流程整合

無論選擇哪種方案，請務必建立 secrets rotation 機制，並定期審查誰有權限存取這些敏感資料。GitOps 讓我們的基礎設施更透明，但這份透明不應該犧牲安全性。
