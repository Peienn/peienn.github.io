# 微服務架構中的服務間通訊：同步 vs 非同步模式的選擇策略

## 前言

當團隊從單體架構遷移到微服務架構後，最常遇到的挑戰之一就是：服務之間該如何通訊？選擇 HTTP REST API 直接呼叫，還是透過 Message Queue 非同步處理？這個決策看似簡單，卻會深刻影響系統的可靠性、效能與維運複雜度。

身為 SRE，我經常看到團隊在初期全部採用同步 HTTP 呼叫，結果在流量高峰時發生連鎖故障；也見過團隊過度使用非同步模式，導致除錯困難、資料一致性問題叢生。本文將分享如何根據實際場景選擇合適的通訊模式，並提供具體的實作建議。

## 同步通訊：簡單直覺但隱藏風險

同步通訊最常見的實作是 HTTP/REST 或 gRPC。服務 A 呼叫服務 B，等待回應後才繼續執行。這種模式直覺易懂，但在微服務環境中容易產生級聯故障（Cascading Failure）。

假設你有一個訂單服務需要呼叫庫存服務：

```python
# 典型的同步呼叫 - 存在風險
import requests

def create_order(order_data):
    # 如果 inventory-service 回應緩慢或掛掉，整個請求就卡住
    response = requests.post(
        "http://inventory-service/api/reserve",
        json={"product_id": order_data["product_id"], "quantity": 1},
        timeout=5  # 至少要設定 timeout！
    )
    if response.status_code != 200:
        raise Exception("庫存保留失敗")
    return save_order(order_data)
```

**適用場景**：需要即時回應、資料強一致性、簡單的請求-回應模式。

**必要防護措施**：務必實作 Circuit Breaker 模式，推薦使用 Resilience4j 或 Python 的 `pybreaker`：

```yaml
# Kubernetes 中搭配 Istio 設定 Circuit Breaker
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: inventory-service
spec:
  host: inventory-service
  trafficPolicy:
    connectionPool:
      http:
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 60s
```

## 非同步通訊：解耦與彈性的代價

非同步模式透過 Message Broker（如 RabbitMQ、Apache Kafka、AWS SQS）實現服務解耦。生產者發送訊息後即可返回，消費者獨立處理。

```python
# 使用 Kafka 的非同步通訊範例
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers=['kafka:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    acks='all',  # 確保訊息持久化
    retries=3
)

def create_order_async(order_data):
    # 訂單建立後，發送事件讓其他服務處理
    order = save_order(order_data)
    producer.send('order-events', {
        'event_type': 'ORDER_CREATED',
        'order_id': order.id,
        'product_id': order_data['product_id'],
        'timestamp': datetime.utcnow().isoformat()
    })
    producer.flush()  # 確保訊息送出
    return order  # 立即回應用戶，不等待庫存處理
```

**適用場景**：跨服務的事件通知、可容忍最終一致性、需要削峰填谷的高流量場景。

**代價**：除錯複雜度增加，需要額外監控訊息積壓與消費延遲。

## 混合模式：實務中的最佳實踐

真實世界中，多數系統會混合使用兩種模式。關鍵在於根據業務需求判斷：

| 判斷維度 | 選擇同步 | 選擇非同步 |
|---------|---------|-----------|
| 回應時間要求 | 用戶需要立即知道結果 | 可接受延遲處理 |
| 一致性要求 | 強一致性 | 最終一致性可接受 |
| 故障容忍度 | 下游故障需立即反映 | 下游故障不應影響主流程 |
| 流量特性 | 穩定可預測 | 有明顯波峰波谷 |

以電商訂單為例的混合設計：

```
用戶下單 
  → [同步] 驗證用戶身份、檢查基本資料
  → [同步] 扣款（金流必須即時確認）
  → [非同步] 發送 ORDER_PAID 事件
      → 庫存服務消費事件，扣減庫存
      → 通知服務消費事件，發送確認信
      → 物流服務消費事件，建立出貨單
```

## 監控與可觀測性：不可忽視的配套

無論選擇哪種模式，完善的可觀測性是維運關鍵。針對非同步通訊，特別需要監控：

```yaml
# Prometheus 告警規則範例 - 監控 Kafka Consumer Lag
groups:
  - name: kafka-alerts
    rules:
      - alert: HighConsumerLag
        expr: kafka_consumer_group_lag > 10000
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Consumer {{ $labels.consumer_group }} lag 過高"
          description: "Topic {{ $labels.topic }} 積壓 {{ $value }} 筆訊息"
```

同時，務必在訊息中加入 Correlation ID，串連整個請求鏈路：

```python
# 在訊息中傳遞追蹤資訊
message = {
    'correlation_id': request.headers.get('X-Correlation-ID', str(uuid4())),
    'event_type': 'ORDER_CREATED',
    'payload': order_data
}
```

## 結論與行動建議

服務間通訊模式的選擇沒有標準答案，但有明確的判斷框架。建議採取以下行動：

1. **盤點現有服務呼叫**：繪製服務依賴圖，標註每個呼叫的一致性與即時性需求
2. **同步呼叫必加防護**：所有 HTTP 呼叫都要設定 timeout、retry 與 circuit breaker
3. **漸進導入非同步**：從通知類、報表類
