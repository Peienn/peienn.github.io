# MySQL 慢查詢排查實戰：從定位到優化的完整指南

## 前言

在維運 MySQL 資料庫時，慢查詢（Slow Query）是最常見也最令人頭痛的問題之一。一條未優化的 SQL 可能在流量高峰時拖垮整個系統，導致連線數爆滿、回應時間暴增，甚至引發連鎖性的服務中斷。

對於中階 DevOps 工程師來說，掌握系統性的慢查詢排查流程至關重要。本文將帶你從啟用慢查詢日誌、分析問題 SQL、到實際優化策略，建立一套可重複運用的排查方法論。

## 啟用與設定慢查詢日誌

MySQL 內建的慢查詢日誌是排查效能問題的第一道防線。首先確認目前的設定狀態：

```sql
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';
```

若尚未啟用，可以動態開啟而不需重啟服務：

```sql
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
SET GLOBAL long_query_time = 1;  -- 超過 1 秒視為慢查詢
SET GLOBAL log_queries_not_using_indexes = 'ON';  -- 記錄未使用索引的查詢
```

建議在 `/etc/mysql/mysql.conf.d/mysqld.cnf` 中永久設定：

```ini
[mysqld]
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1
log_queries_not_using_indexes = 1
```

生產環境中，`long_query_time` 可視業務特性調整，API 服務通常設為 0.5 秒，批次作業系統則可放寬至 5 秒。

## 使用 mysqldumpslow 與 pt-query-digest 分析日誌

當慢查詢日誌累積到一定量後，直接閱讀原始 log 效率極低。MySQL 內建的 `mysqldumpslow` 可快速彙整：

```bash
# 列出出現次數最多的 10 條慢查詢
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

# 列出平均執行時間最長的查詢
mysqldumpslow -s at -t 10 /var/log/mysql/slow.log
```

若需要更強大的分析能力，推薦使用 Percona Toolkit 的 `pt-query-digest`：

```bash
# 安裝 Percona Toolkit
apt-get install percona-toolkit

# 產生詳細的分析報告
pt-query-digest /var/log/mysql/slow.log > slow_report.txt

# 只分析最近一小時的查詢
pt-query-digest --since '1h' /var/log/mysql/slow.log
```

`pt-query-digest` 會將相似的 SQL 正規化後分組統計，輸出包含執行次數、總耗時、鎖定時間等關鍵指標，讓你快速定位最該優先處理的問題查詢。

## 使用 EXPLAIN 深入分析執行計畫

找出問題 SQL 後，下一步是理解 MySQL 如何執行它。`EXPLAIN` 指令是不可或缺的診斷工具：

```sql
EXPLAIN SELECT * FROM orders 
WHERE customer_id = 12345 
AND created_at > '2026-01-01'
ORDER BY created_at DESC;
```

重點觀察欄位：

- **type**：連接類型，從優到劣依序為 `const` > `eq_ref` > `ref` > `range` > `index` > `ALL`。若出現 `ALL` 代表全表掃描
- **key**：實際使用的索引，若為 `NULL` 表示未使用索引
- **rows**：預估掃描的列數，數值越大效能越差
- **Extra**：若出現 `Using filesort` 或 `Using temporary` 需特別留意

MySQL 8.0 以上可使用 `EXPLAIN ANALYZE` 取得實際執行統計：

```sql
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 12345;
```

## 常見優化策略與索引調整

根據 `EXPLAIN` 的分析結果，以下是最常見的優化方向：

**建立複合索引**：針對 WHERE 與 ORDER BY 的欄位組合

```sql
-- 針對上述查詢建立複合索引
CREATE INDEX idx_customer_created ON orders(customer_id, created_at);
```

**避免 SELECT \***：只取需要的欄位，減少 I/O 與網路傳輸

```sql
SELECT order_id, total_amount, created_at FROM orders WHERE customer_id = 12345;
```

**改寫 OR 條件**：多個 OR 可能導致索引失效，考慮改用 UNION

```sql
-- 較差
SELECT * FROM products WHERE category = 'A' OR category = 'B';

-- 較佳
SELECT * FROM products WHERE category IN ('A', 'B');
```

**分頁查詢優化**：深分頁時避免 OFFSET，改用游標式分頁

```sql
-- 效能較差的深分頁
SELECT * FROM logs ORDER BY id LIMIT 1000000, 100;

-- 使用游標分頁
SELECT * FROM logs WHERE id > 1000000 ORDER BY id LIMIT 100;
```

## 結論與行動建議

慢查詢排查並非一次性任務，而是需要持續監控的日常工作。建議採取以下行動：

1. **立即**：確認生產環境已啟用慢查詢日誌，設定合理的閾值
2. **每週**：使用 `pt-query-digest` 產生報告，追蹤 Top 10 問題 SQL
3. **持續**：將慢查詢監控整合至 Grafana 或 Prometheus，設定告警閾值
4. **長期**：建立 Code Review 機制，要求新 SQL 必須附上 EXPLAIN 分析

掌握這套排查流程後，你將能從容應對大多數 MySQL 效能問題，把救火時間轉化為系統性的效能提升。
