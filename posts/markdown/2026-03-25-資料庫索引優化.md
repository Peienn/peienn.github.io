# 當資料庫查詢突然變慢：用 EXPLAIN 診斷並修復低效索引的實戰指南

## 前言

你是否遇過這樣的情況：某個 API 原本回應時間只需要 50ms，某天突然飆升到 3 秒以上？當你檢查程式碼發現沒有任何變更，問題很可能出在資料庫索引上。

隨著資料量成長，原本「堪用」的索引可能逐漸失效。資料分布改變、查詢條件調整、甚至 PostgreSQL 的統計資訊過時，都可能讓查詢優化器選擇錯誤的執行計畫。身為 DevOps 工程師，快速診斷並修復這類問題是維護系統穩定性的關鍵技能。

本文將以 PostgreSQL 為例，透過 `EXPLAIN ANALYZE` 帶你實戰診斷低效索引，並提供具體的優化策略。

## 認識 EXPLAIN ANALYZE：你的第一個診斷工具

`EXPLAIN ANALYZE` 不只顯示查詢計畫，還會實際執行查詢並回報真實的執行時間與資源消耗。這是判斷索引是否有效的最直接方式。

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) 
SELECT * FROM orders 
WHERE customer_id = 12345 
  AND status = 'pending' 
  AND created_at > '2026-01-01';
```

執行後你會看到類似以下輸出：

```
Seq Scan on orders  (cost=0.00..45892.00 rows=1523 width=128) (actual time=892.451..2341.221 rows=47 loops=1)
  Filter: ((customer_id = 12345) AND (status = 'pending'::text) AND (created_at > '2026-01-01'::date))
  Rows Removed by Filter: 2847953
  Buffers: shared hit=12847 read=8234
Planning Time: 0.125 ms
Execution Time: 2341.892 ms
```

關鍵警訊是 `Seq Scan`（全表掃描）以及 `Rows Removed by Filter` 高達 284 萬筆。這代表資料庫掃描了整張表，卻只找到 47 筆符合條件的資料——效率極差。

## 診斷索引未被使用的三大原因

### 原因一：索引根本不存在或欄位順序錯誤

首先確認索引是否存在：

```sql
SELECT indexname, indexdef 
FROM pg_indexes 
WHERE tablename = 'orders';
```

若發現只有 `customer_id` 的單欄索引，而查詢條件包含 `status` 和 `created_at`，PostgreSQL 可能判斷使用索引後還需要大量回表查詢，不如直接全表掃描。

### 原因二：統計資訊過時

PostgreSQL 依賴統計資訊來估算查詢成本。當大量資料異動後，統計資訊可能不準確：

```sql
-- 查看表的統計資訊更新時間
SELECT relname, last_analyze, last_autoanalyze 
FROM pg_stat_user_tables 
WHERE relname = 'orders';

-- 手動更新統計資訊
ANALYZE orders;
```

### 原因三：資料分布導致索引選擇性過低

當某個欄位的值分布極度不均（例如 95% 的訂單都是 `completed` 狀態），查詢 `status = 'completed'` 時，優化器可能認為使用索引沒有意義。

## 建立複合索引的正確策略

針對原本的查詢，我們需要建立一個複合索引。索引欄位的順序至關重要——遵循「等值條件優先、範圍條件在後」原則：

```sql
-- 建立複合索引（使用 CONCURRENTLY 避免鎖表）
CREATE INDEX CONCURRENTLY idx_orders_customer_status_created 
ON orders (customer_id, status, created_at);
```

再次執行 `EXPLAIN ANALYZE`：

```
Index Scan using idx_orders_customer_status_created on orders  (cost=0.56..127.34 rows=47 width=128) (actual time=0.089..0.412 rows=47 loops=1)
  Index Cond: ((customer_id = 12345) AND (status = 'pending'::text) AND (created_at > '2026-01-01'::date))
  Buffers: shared hit=51
Planning Time: 0.198 ms
Execution Time: 0.467 ms
```

執行時間從 2341ms 降至 0.47ms，提升了約 5000 倍！

## 持續監控：建立索引健康度檢查機制

優化不是一次性工作。建議將以下查詢加入監控系統，定期檢查索引使用狀況：

```sql
-- 找出未被使用的索引（可能是多餘索引，浪費空間與寫入效能）
SELECT schemaname, relname, indexrelname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0 
  AND indexrelname NOT LIKE 'pg_%'
ORDER BY pg_relation_size(indexrelid) DESC;

-- 找出缺少索引的慢查詢（需啟用 pg_stat_statements）
SELECT query, calls, mean_exec_time, rows
FROM pg_stat_statements
WHERE mean_exec_time > 1000  -- 平均執行超過 1 秒
ORDER BY mean_exec_time DESC
LIMIT 10;
```

你也可以搭配 Grafana + Prometheus 的 `postgres_exporter`，建立索引掃描次數、Buffer Hit Rate 等儀表板，讓問題在影響使用者前就被發現。

## 結論與行動建議

資料庫索引優化的核心在於「用數據說話」。當遇到查詢效能問題時：

1. **先診斷**：使用 `EXPLAIN ANALYZE` 確認是否為索引問題
2. **找根因**：檢查索引是否存在、統計資訊是否過時、資料分布是否改變
3. **正確建立索引**：注意欄位順序，使用 `CONCURRENTLY` 避免服務中斷
4. **持續監控**：將索引健康度納入日常監控流程

建議你今天就挑選一個生產環境中最慢的查詢，用本文的方法診斷一次。你會發現，很多效能問題的解法比想像中簡單。
