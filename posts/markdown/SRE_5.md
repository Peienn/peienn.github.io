# 可靠性設計（Reliability Engineering）

> "Everything fails, all the time." — Werner Vogels, AWS CTO

系統不會永遠正常運作，問題只在於**什麼時候壞**、**壞了多嚴重**、**多快能恢復**。

可靠性設計的目標不是消滅失敗，而是讓失敗發生時，系統仍然能對用戶提供有意義的服務。

---

## 衡量可靠性的兩個指標

在討論如何設計之前，先定義怎麼量化可靠性。

**MTBF（Mean Time Between Failures）平均故障間隔**
兩次故障之間平均間隔多久，越長代表系統越穩定。

**MTTR（Mean Time To Recovery）平均恢復時間**
故障發生到恢復正常平均花多久，越短代表系統恢復能力越強。

```
可用性 = MTBF / (MTBF + MTTR)

MTBF = 720 小時，MTTR = 1 小時
→ 可用性 = 720 / 721 = 99.86%
```

SRE 通常優先投資**降低 MTTR** 而不是消滅故障，因為完全避免故障是不切實際的，但快速恢復是可以做到的。

---

## 可靠性設計的三個層次

```
層次一：避免失敗（Fault Prevention）
  好的程式碼、充分測試、漸進式部署
  → 開發工程師的責任

層次二：容忍失敗（Fault Tolerance）
  元件掛掉時，系統仍能部分運作
  → SRE 的核心工作

層次三：從失敗中恢復（Fault Recovery）
  快速偵測、自動恢復、Postmortem
  → SRE 的核心工作
```

## Blast Radius（爆炸半徑）

每次設計系統時，都要問一個問題：

> **這個元件掛掉，會影響多少功能？**

影響範圍越大，blast radius 越大，風險越高。

```
Blast radius 小（好）：
  AI 分析服務掛掉 → 只影響 AI 分析功能
  其他功能完全不受影響

Blast radius 大（危險）：
  共用的 Redis 掛掉 → Session、Cache、Pub/Sub 同時壞
  三個功能同時受影響
```

縮小 blast radius 是可靠性設計的第一步，做法是**隔離**：讓不同功能依賴不同的元件，避免單點故障影響範圍過大。

## 核心概念一：Circuit Breaker（熔斷器）

### 為什麼需要

一個慢的下游服務，可能把整個上游都拖垮，這叫做**級聯故障（Cascading Failure）**。

```
下游服務回應變慢（例如 DB query 要 30 秒）
  → 上游等待 timeout，連線池被佔滿
  → 新請求也開始排隊
  → 整個服務癱瘓
```

### 三個狀態

```
Closed（正常）
  所有請求正常通過，持續計算錯誤率
        │
        │ 錯誤率超過閾值
        ▼
Open（熔斷）
  直接拒絕所有請求，立刻返回錯誤
  不等待，不讓請求堆積
        │
        │ 等待一段時間後
        ▼
Half-Open（探測）
  放少量請求進去測試
  成功 → 回到 Closed
  失敗 → 繼續 Open
```

### 關鍵參數

- **失敗閾值**：錯誤率達到多少才熔斷（例如 50%）
- **熔斷時間**：Open 狀態維持多久才進入 Half-Open（例如 30 秒）
- **探測請求數**：Half-Open 時放幾個請求進去測試

Circuit Breaker 通常搭配 **Fallback** 使用：熔斷時不是直接返回錯誤，而是切換到備援來源，例如從 DB fallback 到 Cache。

## 核心概念二：Retry with Backoff（重試機制）

### 為什麼需要

網路閃斷、服務短暫抖動，是分散式系統的常態。
這類錯誤通常很快就會自行恢復，立刻重試往往就能成功。

### Exponential Backoff（指數退避）

不要立刻重試，每次重試間隔越來越長：

```
第一次失敗 → 等 100ms → 重試
第二次失敗 → 等 200ms → 重試
第三次失敗 → 等 400ms → 重試
第四次失敗 → 放棄，返回錯誤
```

### Jitter（隨機抖動）

如果大量請求同時失敗又同時重試，會再次打垮服務。
在退避時間上加入隨機抖動，讓重試時間分散，避免這種情況。

### 什麼情況不該 Retry

| 錯誤類型 | 適合 Retry？ |
|---|---|
| 網路閃斷、timeout | ✅ |
| 5xx Server Error | ✅ |
| 4xx Client Error（Bad Request）| ❌ 重試沒有意義 |
| 資料衝突（409 Conflict）| ❌ 重試只會繼續衝突 |
| 冪等操作（查詢）| ✅ 安全 |
| 非冪等操作（扣款、發送訊息）| ⚠️ 需要去重機制才能 Retry |

## 核心概念三：Timeout（超時設定）

### 為什麼需要

沒有 Timeout 的服務，遇到慢的下游時會無限等待，最終資源耗盡。
Timeout 是 Circuit Breaker 的前提條件：沒有 Timeout，Circuit Breaker 永遠不會觸發。

### 三個層次

```
連線層 Timeout：建立 TCP 連線的最大等待時間
請求層 Timeout：整個 HTTP 請求的最大等待時間
業務層 Timeout：特定操作（DB query、外部 API）的最大等待時間
```

### 設定原則

- 不要設太長：太長等於沒有設
- 不要設太短：太短會產生大量誤報
- 參考 p99：以實際回應時間的 p99 為基準，設為 2～3 倍

## 核心概念四：Graceful Degradation（優雅降級）

### 概念

當部分功能無法使用時，系統仍然提供**核心功能**，而不是整個服務直接掛掉。

```
功能 C 的依賴服務掛掉
  ↓
功能 A ✅  功能 B ✅  功能 C ❌

而不是：

整個服務 ❌
```

### Fallback 的層次

```
第一層：從 Cache 讀取（最快，但資料可能不是最新）
第二層：從主要資料庫讀取（較慢，但資料最新）
第三層：返回靜態預設值（最後手段，但比完全失敗好）
```

設計 Graceful Degradation 的關鍵是**提前決定好降級策略**，不要在事故發生時才臨時想。

---

## 核心概念五：Bulkhead（艙壁隔離）

### 概念

船艙設計的靈感：把船分成多個密封艙，一個艙進水不會導致整艘船沉沒。

```
沒有 Bulkhead：
  一個功能佔用大量資源（連線池、執行緒）
    → 所有功能共享的資源被耗盡
    → 其他功能也跟著受影響

有 Bulkhead：
  每個功能有獨立的資源池，有各自的上限
    → 一個功能再慢，不影響其他功能
```

### 常見做法

- 不同功能使用**獨立的連線池**，設定各自的上限
- 不同優先級的請求使用**不同的執行緒池**或 Queue
- 非核心服務（例如 AI 分析）和核心服務（例如即時訊息）使用**完全獨立的服務**

---

## 五個概念的關係

這五個概念不是獨立的，通常組合使用：

```
Timeout
  → 確保不會無限等待
  → 讓 Circuit Breaker 能正確計算錯誤率

Circuit Breaker
  → 偵測到錯誤率過高時熔斷
  → 觸發 Fallback

Graceful Degradation（Fallback）
  → Circuit Breaker 觸發後的降級策略
  → 確保核心功能仍然可用

Retry + Backoff
  → 短暫抖動時的自動恢復機制
  → 在 Circuit Breaker 觸發前先嘗試

Bulkhead
  → 確保一個功能的失敗不會蔓延到其他功能
  → 從架構層面縮小 blast radius
```

## 可靠性設計的取捨

可靠性設計不是免費的：

| 設計 | 解決的問題 | 代價 |
|---|---|---|
| Circuit Breaker | 防止級聯故障 | 程式碼複雜度增加 |
| Retry + Backoff | 容忍短暫抖動 | 整體延遲可能增加 |
| Timeout | 防止無限等待 | 需要精確校準時間 |
| Graceful Degradation | 部分故障仍可用 | 需要設計多個 fallback 路徑 |
| Bulkhead | 故障隔離，不蔓延 | 資源使用率下降 |

SRE 的工作是在**可靠性和複雜度之間找平衡**。過度設計和設計不足都是問題。

---

## 小結

可靠性設計的思路框架：

```
1. 找出 blast radius 最大的元件 → 優先隔離
2. 設計 fallback 路徑 → 確保核心功能不中斷
3. 加入 Timeout + Circuit Breaker → 防止故障蔓延
4. 對非冪等操作謹慎使用 Retry → 避免重複副作用
5. 用 MTTR / MTBF 量化可靠性 → 持續追蹤改善
```

好的可靠性設計，讓系統在部分元件失敗時，仍然能對用戶提供有意義的服務。
