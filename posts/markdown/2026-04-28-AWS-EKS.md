# 解決 AWS EKS Pod 跨可用區（AZ）流量成本過高的實戰技巧

## 前言

在 AWS EKS 上運行生產環境時，許多團隊會發現每月帳單中有一筆意外的高額費用——跨可用區（Cross-AZ）的資料傳輸費用。這個問題特別容易被忽略，因為 Kubernetes 預設的 Service 負載均衡機制會將流量隨機分配到所有健康的 Pod，不論它們位於哪個 AZ。

當你的叢集規模擴大、微服務之間的呼叫頻繁時，這筆費用可能佔據網路成本的 30% 以上。本文將介紹如何透過 Topology Aware Routing 與相關配置，有效降低這筆隱藏成本。

## 理解跨 AZ 流量的產生原因

在典型的 EKS 多 AZ 部署中，假設你有一個 Service 後面掛載 6 個 Pod，平均分佈在 3 個 AZ。當 Pod A（位於 us-east-1a）呼叫該 Service 時，有 2/3 的機率流量會被導向其他 AZ 的 Pod。

AWS 對跨 AZ 流量收費為每 GB 約 $0.01（雙向計算則為 $0.02），看似便宜，但當內部服務每日傳輸數 TB 資料時，月費用可輕易突破數千美元。

檢視目前的跨 AZ 流量狀況，可以使用 VPC Flow Logs 搭配 Athena 查詢：

```sql
SELECT az_id, SUM(bytes) as total_bytes
FROM vpc_flow_logs
WHERE src_addr LIKE '10.%' AND dst_addr LIKE '10.%'
AND date = current_date - interval '1' day
GROUP BY az_id
ORDER BY total_bytes DESC;
```

## 啟用 Topology Aware Routing

Kubernetes 1.23+ 開始正式支援 Topology Aware Routing（前身為 Topology Aware Hints）。這個功能讓 kube-proxy 優先將流量導向同一個 AZ 內的 Pod。

在 EKS 上啟用此功能，只需在 Service 加上對應的 annotation：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-backend-service
  annotations:
    service.kubernetes.io/topology-mode: Auto
spec:
  selector:
    app: my-backend
  ports:
    - port: 80
      targetPort: 8080
```

啟用後，可以透過以下指令確認 EndpointSlice 是否正確產生 hints：

```bash
kubectl get endpointslices -l kubernetes.io/service-name=my-backend-service -o yaml | grep -A 5 "hints"
```

預期輸出應包含類似 `forZones` 的欄位，指示該 endpoint 偏好接收來自特定 zone 的流量。

## 確保 Pod 分佈均衡

Topology Aware Routing 有一個重要前提：各 AZ 的 Pod 數量必須相對均衡。若某個 AZ 只有 1 個 Pod 而其他有 5 個，Kubernetes 會自動停用該 Service 的拓撲感知功能以避免負載不均。

使用 Pod Topology Spread Constraints 確保部署均勻：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-backend
spec:
  replicas: 6
  template:
    spec:
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: my-backend
      containers:
        - name: app
          image: my-backend:latest
```

`maxSkew: 1` 確保各 AZ 之間的 Pod 數量差異不超過 1，這對維持 Topology Aware Routing 的正常運作至關重要。

## 監控與驗證成效

部署完成後，建議設置監控來追蹤跨 AZ 流量的變化。可以使用 Prometheus 搭配 kube-proxy 的 metrics：

```yaml
# prometheus-scrape-config.yaml
- job_name: 'kube-proxy'
  kubernetes_sd_configs:
    - role: pod
  relabel_configs:
    - source_labels: [__meta_kubernetes_pod_label_k8s_app]
      regex: kube-proxy
      action: keep
```

另外，建議部署一個簡單的測試工具來驗證流量是否確實留在同一 AZ：

```bash
# 在特定 AZ 的 Pod 中執行
kubectl exec -it test-pod -- sh -c "for i in \$(seq 1 100); do curl -s my-backend-service/health | grep zone; done | sort | uniq -c"
```

理想狀況下，90% 以上的回應應來自同一個 AZ。

## 結論與行動建議

跨 AZ 流量成本是 EKS 營運中容易被忽略但影響顯著的議題。透過本文介紹的方法，你可以：

1. **立即行動**：為關鍵的內部 Service 啟用 `topology-mode: Auto` annotation
2. **短期優化**：檢視所有 Deployment，加入 Topology Spread Constraints 確保 Pod 分佈均衡
3. **長期監控**：建立 VPC Flow Logs 分析 pipeline，持續追蹤跨 AZ 流量佔比

實務經驗顯示，這些優化通常能降低 40-60% 的內部網路傳輸費用，對於中大型叢集而言，每月可節省數百至數千美元。現在就開始檢視你的 EKS 叢集，找出那些高頻呼叫的內部服務，逐步套用這些最佳實踐吧。
