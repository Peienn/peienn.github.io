# Helm Chart 版本升級踩雷指南：三個常見陷阱與避坑技巧

## 前言

當你在 Kubernetes 環境中使用 Helm 管理應用程式部署時，Chart 版本升級往往是最容易出問題的環節。相信不少人都有過這樣的經驗：明明只是執行一個看似簡單的 `helm upgrade`，結果卻導致服務中斷、配置遺失，甚至整個 Release 進入無法回復的狀態。

這篇文章將分享三個在 Helm 升級過程中最常遇到的陷阱，以及對應的預防與解決策略。這些經驗來自實際的生產環境事故，希望能幫助你在下次升級前做好萬全準備。

## 陷阱一：Values 被意外覆蓋的慘劇

這是新手最常踩到的雷。當你使用 `helm upgrade` 時，如果沒有帶上 `--reuse-values` 參數，Helm 預設會使用 Chart 的預設值來覆蓋現有配置。

假設你原本部署時設定了 `replicas: 5`，但升級時忘記帶上原本的 values：

```bash
# 危險操作：可能會覆蓋你的自訂配置
helm upgrade my-app bitnami/nginx --version 15.0.0
```

正確的做法應該是：

```bash
# 方法一：重複使用現有 values
helm upgrade my-app bitnami/nginx --version 15.0.0 --reuse-values

# 方法二：明確指定 values 檔案（推薦）
helm upgrade my-app bitnami/nginx --version 15.0.0 -f values-production.yaml
```

**最佳實踐**：將你的 values 檔案納入 Git 版本控制，每次升級都明確指定 values 檔案，而非依賴 `--reuse-values`。這樣可以確保配置的可追溯性。

## 陷阱二：CRD 升級導致的不相容問題

Helm 對 CRD（Custom Resource Definition）的處理機制相當特殊：**Helm 只會在首次安裝時建立 CRD，升級時不會更新它們**。這個設計是為了避免意外刪除使用者已建立的 Custom Resources。

但這也帶來一個問題：當新版 Chart 需要更新後的 CRD 才能正常運作時，升級就會失敗。

```bash
# 檢查現有 CRD 版本
kubectl get crd certificates.cert-manager.io -o jsonpath='{.spec.versions[*].name}'

# 手動更新 CRD（以 cert-manager 為例）
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.14.0/cert-manager.crds.yaml

# 確認 CRD 更新後再升級 Chart
helm upgrade cert-manager jetstack/cert-manager --version 1.14.0 -n cert-manager
```

升級前，務必閱讀 Chart 的 CHANGELOG 或 UPGRADE NOTES，確認是否有 CRD 變更需要手動處理。

## 陷阱三：Hook 執行失敗卡住整個 Release

Helm Hooks 是強大的功能，可以在升級前後執行資料庫遷移、備份等任務。但如果 Hook Job 執行失敗，整個 Release 會卡在 `pending-upgrade` 狀態，進退兩難。

```bash
# 查看卡住的 Release 狀態
helm list -a
# NAME     STATUS          
# my-app   pending-upgrade

# 查看失敗的 Hook Job
kubectl get jobs -l "helm.sh/hook"
```

解決方式：

```bash
# 強制標記升級失敗，回復到上一個版本
helm rollback my-app

# 如果 rollback 也失敗，可能需要手動清理
kubectl delete job my-app-pre-upgrade-hook
helm upgrade my-app ./my-chart --force --cleanup-on-fail
```

**預防建議**：在 Hook 定義中加入適當的超時與刪除策略：

```yaml
annotations:
  "helm.sh/hook": pre-upgrade
  "helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded
  "helm.sh/hook-weight": "-5"
```

## 升級前的安全檢查清單

在每次執行 `helm upgrade` 之前，建議執行以下檢查：

```bash
# 1. 先用 diff 插件預覽變更
helm diff upgrade my-app ./my-chart -f values.yaml

# 2. 執行 dry-run 驗證
helm upgrade my-app ./my-chart -f values.yaml --dry-run --debug

# 3. 確認可以回滾
helm history my-app
```

安裝 helm-diff 插件：

```bash
helm plugin install https://github.com/databus23/helm-diff
```

## 結論與行動建議

Helm 升級問題大多可以透過事前準備來避免。總結三個核心原則：

1. **永遠使用版本控制的 values 檔案**，不要依賴 `--reuse-values`
2. **升級前閱讀 CHANGELOG**，特別注意 CRD 和 Breaking Changes
3. **善用 `helm diff` 和 `--dry-run`**，先預覽再執行

建議你今天就在測試環境中實際操作一次升級流程，熟悉這些工具和參數。當生產環境真的需要緊急升級時，你會感謝自己的準備。
