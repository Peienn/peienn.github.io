# 資料庫連線池爆滿？從根本解決 Connection Pool Exhaustion 的五個實戰策略

## 前言

凌晨三點，你被 PagerDuty 叫醒，看到滿屏的 `Cannot acquire connection from pool` 錯誤。應用程式完全無法回應，而資料庫的 CPU 使用率卻異常地低。這不是資料庫掛了，而是連線池被耗盡了。

Connection Pool Exhaustion（連線池耗盡）是中大型系統最常見的效能瓶頸之一。它的棘手之處在於：問題往往不在資料庫本身，而是藏在應用程式的連線管理邏輯中。本文將從實戰角度，分享五個診斷與解決連線池爆滿的具體策略。

## 理解連線池的生命週期與監控指標

在解決問題之前，你需要先能「看見」問題。大多數連線池實作（如 HikariCP、PgBouncer、c3p0）都提供監控指標，關鍵是知道該看哪些數值。

以 Java 生態系最常用的 HikariCP 為例，透過 JMX 或 Micrometer 可以取得以下關鍵指標：

```yaml
# Prometheus 監控的關鍵指標
hikaricp_connections_active      # 目前使用中的連線數
hikaricp_connections_idle        # 閒置連線數
hikaricp_connections_pending     # 等待取得連線的執行緒數
hikaricp_connections_timeout_total # 連線取得逾時的累計次數
```

當 `pending` 持續大於 0，或 `timeout_total` 開始攀升，就是連線池即將爆滿的警訊。建議在 Grafana 設定告警：當 `active / maximum_pool_size > 0.8` 持續超過 30 秒，立即通知。

## 找出連線洩漏的元兇：開啟洩漏偵測

連線池爆滿最常見的原因是「連線洩漏」——程式碼取得連線後，忘記歸還。HikariCP 提供內建的洩漏偵測機制：

```properties
# application.properties
spring.datasource.hikari.leak-detection-threshold=30000
```

這會在連線被持有超過 30 秒時，在 log 中印出取得該連線的完整 stack trace：

```
[WARN] Connection leak detection triggered for connection com.mysql.cj.jdbc.ConnectionImpl@3a5ed7a6
    at com.example.UserService.findUserById(UserService.java:45)
    at com.example.OrderController.createOrder(OrderController.java:78)
```

有了 stack trace，你就能精準定位是哪段程式碼沒有正確關閉連線。常見的問題模式包括：

- 在 `try` 區塊取得連線，卻沒有用 `try-with-resources` 或 `finally` 確保關閉
- 在迴圈中重複取得連線，而非重複使用同一個連線
- 使用 ORM 時，手動開啟的 Session 沒有正確關閉

## 調整連線池大小：不是越大越好

很多人遇到連線池爆滿的第一反應是「把 pool size 調大」，這往往是錯誤的方向。連線池的最佳大小遵循一個經驗公式：

```
pool_size = (core_count * 2) + effective_spindle_count
```

對於 SSD 儲存的系統，一台 4 核心的應用伺服器，連線池大小設定在 10-15 通常就足夠。盲目調大反而會導致資料庫端的連線管理開銷增加，造成整體效能下降。

以 PostgreSQL 為例，檢查資料庫端的連線狀態：

```sql
SELECT state, count(*) 
FROM pg_stat_activity 
WHERE datname = 'your_database'
GROUP BY state;
```

如果看到大量 `idle` 狀態的連線，代表應用端的 pool size 可能設定過大。

## 導入 PgBouncer 作為連線池中介層

當你的架構包含多個應用實例（例如 Kubernetes 中的多個 Pod），每個實例都維護自己的連線池，資料庫端的連線數會爆炸性成長。此時應該考慮導入連線池中介層。

```bash
# 安裝 PgBouncer
apt-get install pgbouncer

# /etc/pgbouncer/pgbouncer.ini 關鍵設定
[databases]
mydb = host=127.0.0.1 port=5432 dbname=mydb

[pgbouncer]
pool_mode = transaction    # 交易層級的連線共享
max_client_conn = 1000     # 接受的最大客戶端連線
default_pool_size = 20     # 每個資料庫的連線池大小
```

`pool_mode = transaction` 是關鍵設定，它讓連線在每個交易結束後就歸還池中，大幅提高連線利用率。但要注意，這模式下 prepared statements 和 session 變數會有限制。

## 實作優雅降級與背壓機制

即使做了所有優化，流量尖峰時連線池仍可能飽和。與其讓請求無限等待，不如實作優雅降級：

```java
// HikariCP 設定合理的等待逾時
spring.datasource.hikari.connection-timeout=3000  // 3秒

// 在程式碼中處理逾時例外
@ExceptionHandler(SQLTransientConnectionException.class)
public ResponseEntity<String> handlePoolExhausted(Exception e) {
    log.warn("Connection pool exhausted, returning 503");
    return ResponseEntity.status(503)
        .header("Retry-After", "5")
        .body("Service temporarily unavailable");
}
```

回傳 503 並帶上 `Retry-After` header，讓前端或 API Gateway 知道該重試，而非讓請求堆積導致雪崩效應。

## 結論與行動建議

連線池爆滿從來不是單一原因造成的，而是監控不足、程式碼缺陷、設定不當三者的綜合結果。建議你從以下步驟開始改善：

1. **今天就做**：開啟 leak detection，加入連線池指標監控
2. **本週完成**：檢視程式碼中所有取得資料庫連線的位置，確保正確關閉
3. **本月評估**：若運行多實例架構，評估導入 PgBouncer 的可行性

記住：好的連線池管理不是讓你的系統「不會爆」，而是讓你在爆之前就能預警，爆的時候能優雅處理，爆之後能快速定位原因。
