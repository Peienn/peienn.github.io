# 非同步任務處理中的重試機制設計：避免雪崩效應的實戰指南

## 前言

在分散式系統中，非同步任務處理是不可或缺的架構元素。無論是發送通知信、處理圖片轉檔，還是同步第三方資料，我們都仰賴 Message Queue 搭配 Worker 來完成這些工作。然而，當下游服務發生故障時，設計不良的重試機制往往會讓小問題演變成系統性災難——任務堆積、Worker 過載、甚至觸發雪崩效應。

本文將深入探討如何設計一個健壯的重試機制，幫助你的非同步任務系統在面對失敗時能優雅地恢復，而非雪上加霜。

## 為什麼簡單的重試策略會害死你

許多工程師在實作重試時，直覺會寫出這樣的程式碼：

```python
# ❌ 危險的重試實作
def process_task(task):
    max_retries = 5
    for attempt in range(max_retries):
        try:
            call_external_api(task)
            return
        except Exception:
            time.sleep(1)  # 固定等待 1 秒後重試
    raise MaxRetriesExceeded()
```

這段程式碼的問題在於：當外部 API 故障時，所有 Worker 會以相同的間隔同時發起重試請求。假設你有 100 個任務同時失敗，它們會在第 1、2、3、4、5 秒時同步發起 100 次請求，這種「重試風暴」會讓已經不穩定的下游服務更難恢復。

## 指數退避與抖動：分散重試壓力

解決重試風暴的標準做法是採用 **Exponential Backoff with Jitter**（指數退避加抖動）。核心概念是讓每次重試的等待時間呈指數增長，並加入隨機因子來分散請求：

```python
import random
import time

def calculate_backoff(attempt: int, base_delay: float = 1.0, max_delay: float = 300.0) -> float:
    """計算帶有 full jitter 的退避時間"""
    exponential_delay = base_delay * (2 ** attempt)
    capped_delay = min(exponential_delay, max_delay)
    # Full Jitter: 在 0 到 capped_delay 之間隨機取值
    return random.uniform(0, capped_delay)

# 使用範例
for attempt in range(5):
    delay = calculate_backoff(attempt)
    print(f"Attempt {attempt}: wait {delay:.2f}s")
    # Attempt 0: wait 0.73s
    # Attempt 1: wait 1.42s
    # Attempt 2: wait 2.89s
    # Attempt 3: wait 7.21s
    # Attempt 4: wait 18.53s
```

這種策略確保重試請求在時間軸上自然分散，大幅降低同時打爆下游服務的機率。

## 使用 Celery 實作生產級重試機制

在 Python 生態系中，Celery 是最常見的非同步任務框架。以下示範如何正確配置重試：

```python
from celery import Celery
from celery.exceptions import MaxRetriesExceededError
import requests

app = Celery('tasks', broker='redis://localhost:6379/0')

@app.task(
    bind=True,
    max_retries=5,
    autoretry_for=(requests.RequestException,),
    retry_backoff=True,        # 啟用指數退避
    retry_backoff_max=600,     # 最大等待 10 分鐘
    retry_jitter=True          # 啟用隨機抖動
)
def sync_user_data(self, user_id: int):
    try:
        response = requests.post(
            'https://api.external-service.com/sync',
            json={'user_id': user_id},
            timeout=10
        )
        response.raise_for_status()
    except requests.RequestException as exc:
        raise self.retry(exc=exc)
```

關鍵配置說明：
- `retry_backoff=True`：自動啟用指數退避
- `retry_jitter=True`：加入隨機因子避免同步重試
- `retry_backoff_max`：設定合理上限，避免等待過久

## 實作 Dead Letter Queue 處理無法恢復的任務

重試終究有次數限制，對於多次重試仍失敗的任務，我們需要 Dead Letter Queue（DLQ）來暫存，而非直接丟棄：

```python
@app.task(bind=True, max_retries=5)
def process_payment(self, order_id: int):
    try:
        result = payment_gateway.charge(order_id)
        return result
    except RecoverableError as exc:
        raise self.retry(exc=exc, countdown=calculate_backoff(self.request.retries))
    except Exception as exc:
        # 無法恢復的錯誤，送入 DLQ
        send_to_dlq.delay({
            'task': 'process_payment',
            'args': {'order_id': order_id},
            'error': str(exc),
            'failed_at': datetime.utcnow().isoformat()
        })
        # 發送告警通知 on-call 工程師
        alert_oncall(f"Payment task failed permanently: {order_id}")
```

搭配監控儀表板追蹤 DLQ 深度，你可以快速識別系統性問題：

```bash
# 使用 redis-cli 監控 DLQ 長度
watch -n 5 'redis-cli LLEN celery-dead-letter-queue'
```

## 結論與行動建議

設計良好的重試機制是非同步系統穩定性的基石。以下是你明天就可以開始執行的改善項目：

1. **審查現有重試邏輯**：檢查是否有使用固定間隔重試，立即加入指數退避與抖動
2. **設定合理的重試上限**：根據業務需求決定，通常 3-5 次配合指數退避已足夠
3. **建立 DLQ 機制**：確保失敗任務有跡可循，而非消失在黑洞中
4. **監控重試指標**：在 Prometheus 中追蹤重試次數與 DLQ 深度，設定告警閾值

記住：重試機制的目標是讓系統自我修復，而非加速崩潰。花時間把這塊做好，你的 PagerDuty 帳單會感謝你的。
