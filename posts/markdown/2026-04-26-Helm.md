# Helm Chart 依賴管理實戰：用 Dependency 優雅處理多服務部署

## 前言

當你的 Kubernetes 應用從單一服務成長為包含 Redis、PostgreSQL、Nginx 等多個元件的微服務架構時，如何有效管理這些相互依賴的 Helm Charts 就成為關鍵課題。許多團隊選擇為每個服務建立獨立的 Helm release，但這往往導致部署順序混亂、版本不一致，以及難以追蹤的設定散落各處。

本文將深入探討 Helm 的 Dependency 機制，示範如何建立一個主 Chart 來統一管理所有子服務，讓你的部署流程更加優雅且可維護。

## 理解 Helm Dependency 的運作原理

Helm Dependency 允許你在 `Chart.yaml` 中宣告對其他 Charts 的依賴關係。當執行 `helm dependency update` 時，Helm 會將所有依賴的 Charts 下載到 `charts/` 目錄中，最終打包成單一可部署的 Chart。

```yaml
# Chart.yaml
apiVersion: v2
name: my-application
version: 1.0.0
dependencies:
  - name: postgresql
    version: "12.5.8"
    repository: "https://charts.bitnami.com/bitnami"
  - name: redis
    version: "17.11.3"
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled
```

執行依賴更新：

```bash
helm dependency update ./my-application
helm dependency list ./my-application
```

這種架構的優勢在於：所有服務版本集中管理、單一指令完成完整部署、設定值統一在同一份 `values.yaml` 中維護。

## 透過 Values 覆寫子 Chart 設定

依賴管理的精髓在於能從父 Chart 的 `values.yaml` 統一控制所有子 Chart 的設定。子 Chart 的設定需放在以該 Chart 名稱為 key 的區塊下：

```yaml
# values.yaml
# 應用本身的設定
replicaCount: 3
image:
  repository: my-app
  tag: v1.2.0

# PostgreSQL 子 Chart 設定
postgresql:
  auth:
    username: appuser
    password: secretpassword
    database: myappdb
  primary:
    persistence:
      size: 20Gi

# Redis 子 Chart 設定（可透過 condition 開關）
redis:
  enabled: true
  architecture: standalone
  auth:
    password: redispassword
```

若需要在父 Chart 的 templates 中引用子 Chart 產生的資源名稱，可使用內建函式：

```yaml
# templates/deployment.yaml
env:
  - name: DATABASE_HOST
    value: "{{ .Release.Name }}-postgresql"
  - name: REDIS_HOST
    value: "{{ .Release.Name }}-redis-master"
```

## 使用 Condition 與 Tags 實現彈性部署

實務上，不同環境可能需要不同的服務組合。開發環境可能使用外部託管資料庫，而測試環境則需要完整部署。Helm 提供 `condition` 和 `tags` 兩種機制來控制依賴的啟用狀態：

```yaml
# Chart.yaml
dependencies:
  - name: postgresql
    version: "12.5.8"
    repository: "https://charts.bitnami.com/bitnami"
    condition: postgresql.enabled
    tags:
      - database
  - name: redis
    version: "17.11.3"
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled
    tags:
      - cache
```

部署時根據環境選擇性啟用：

```bash
# 開發環境：僅部署應用本身
helm install my-app ./my-application \
  --set postgresql.enabled=false \
  --set redis.enabled=false

# 測試環境：完整部署
helm install my-app ./my-application \
  --set-json 'tags={"database":true,"cache":true}'
```

## 處理依賴更新與版本鎖定

團隊協作時，確保所有成員使用相同版本的依賴至關重要。`Chart.lock` 檔案記錄了精確的依賴版本，應納入版本控制：

```bash
# 更新依賴並產生 Chart.lock
helm dependency update ./my-application

# 根據 Chart.lock 重建依賴（CI/CD 環境建議使用）
helm dependency build ./my-application
```

在 CI/CD pipeline 中建議的做法：

```yaml
# .gitlab-ci.yml 範例
deploy:
  script:
    - helm dependency build ./my-application
    - helm upgrade --install my-app ./my-application
      --namespace production
      --values values-production.yaml
      --atomic
      --timeout 10m
```

`--atomic` 參數確保部署失敗時自動回滾，避免依賴服務部分啟動造成的不一致狀態。

## 結論與行動建議

Helm Dependency 機制讓複雜的多服務部署變得可管理且可重現。透過本文介紹的技巧，你可以：

1. **立即行動**：檢視現有專案，將散落的多個 Helm releases 重構為單一帶有依賴的主 Chart
2. **建立規範**：將 `Chart.lock` 納入 Git，確保團隊版本一致性
3. **善用 Condition**：為不同環境設計彈性的依賴啟用策略，避免維護多份 Chart

當依賴關係變得更複雜時，可進一步研究 Helmfile 或 ArgoCD ApplicationSet 來管理多 Chart 部署，但 Helm 原生的 Dependency 機制已足以應付大多數中型專案的需求。
