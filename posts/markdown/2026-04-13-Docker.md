# Docker 多階段建置實戰：將 Node.js 映像檔從 1.2GB 瘦身到 150MB

## 前言

你是否曾經打包完一個簡單的 Node.js 應用程式，結果發現映像檔竟然超過 1GB？這不僅浪費儲存空間，更嚴重的是拖慢 CI/CD pipeline 的部署速度，每次推送都要等待漫長的傳輸時間。

Docker 的多階段建置（Multi-stage Build）正是解決這個痛點的最佳方案。這個功能從 Docker 17.05 開始支援，讓我們能在單一 Dockerfile 中使用多個 `FROM` 指令，將建置環境與執行環境分離，最終只保留應用程式真正需要的檔案。

對於中階 DevOps 工程師來說，掌握多階段建置不只是優化技巧，更是撰寫 Production-ready Dockerfile 的基本功。

## 傳統單階段建置的問題

讓我們先看一個典型的 Node.js Dockerfile：

```dockerfile
FROM node:20
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

這個寫法看起來沒什麼問題，但執行 `docker images` 檢查大小時：

```bash
$ docker images myapp
REPOSITORY   TAG       SIZE
myapp        latest    1.24GB
```

問題出在哪裡？`node:20` 基礎映像檔本身就包含完整的 Debian 系統、npm、yarn、編譯工具鏈等，這些在執行階段根本用不到。更糟的是，`node_modules` 裡面還包含了所有 `devDependencies`。

## 多階段建置的核心概念

多階段建置的精髓在於：**建置階段用完整工具，執行階段只留必要元件**。

改寫後的 Dockerfile：

```dockerfile
# ===== 建置階段 =====
FROM node:20 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# ===== 執行階段 =====
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production

# 只安裝 production dependencies
COPY package*.json ./
RUN npm ci --only=production && npm cache clean --force

# 從 builder 階段複製編譯產物
COPY --from=builder /app/dist ./dist

EXPOSE 3000
USER node
CMD ["node", "dist/index.js"]
```

關鍵語法解析：
- `AS builder`：為階段命名，方便後續引用
- `FROM node:20-alpine`：第二階段使用輕量級 Alpine 版本
- `COPY --from=builder`：跨階段複製檔案，只取需要的產物

## 進階優化技巧

### 善用 .dockerignore

建立 `.dockerignore` 避免不必要的檔案進入建置環境：

```plaintext
node_modules
dist
.git
.env*
*.md
Dockerfile
docker-compose.yml
```

### 考慮 Distroless 映像檔

如果想更進一步壓縮體積，可以使用 Google 的 Distroless 映像檔：

```dockerfile
FROM gcr.io/distroless/nodejs20-debian12 AS runner
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
CMD ["dist/index.js"]
```

Distroless 移除了 shell、套件管理器等工具，大幅減少攻擊面，但也意味著你無法 `docker exec` 進去除錯。

### 驗證優化成果

```bash
$ docker images myapp
REPOSITORY   TAG          SIZE
myapp        single       1.24GB
myapp        multistage   148MB
myapp        distroless   89MB
```

從 1.24GB 降到 148MB，縮減了近 **88%** 的體積！

## 實務注意事項

多階段建置雖然強大，但有幾點需要注意：

1. **快取失效問題**：將 `package*.json` 複製與 `npm install` 放在 `COPY . .` 之前，這樣只有 dependencies 變動時才會重新安裝。

2. **Native modules 相容性**：如果使用 `bcrypt`、`sharp` 等包含原生編譯模組的套件，必須確保建置階段與執行階段的系統架構相容，或在執行階段重新安裝。

3. **除錯便利性**：可保留一個 `debug` target 供開發使用：

```bash
$ docker build --target builder -t myapp:debug .
```

## 結論與行動建議

多階段建置是每位 DevOps 工程師都該熟練的技巧。建議你現在就動手：

1. 檢視專案中現有的 Dockerfile，執行 `docker images` 查看體積
2. 套用多階段建置模式重構 Dockerfile
3. 在 CI/CD pipeline 中觀察建置與部署時間的改善
4. 考慮將 Alpine 或 Distroless 作為執行環境的標準選擇

當映像檔體積縮小，部署速度提升，整個開發流程都會變得更加順暢。這個小小的優化，能為團隊帶來長期可觀的效益。
