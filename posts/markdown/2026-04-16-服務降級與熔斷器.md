# 熔斷器狀態轉換實戰：用 Resilience4j 打造自癒微服務架構

## 前言

在微服務架構中，服務之間的依賴關係錯綜複雜，當某個下游服務出現延遲或故障時，如果沒有適當的防護機制，故障會像滾雪球般蔓延到整個系統。這就是著名的「雪崩效應」（Cascading Failure）。

熔斷器模式（Circuit Breaker Pattern）是解決此問題的關鍵手段，但許多工程師僅停留在「知道要用」的階段，對於熔斷器的三態轉換邏輯、閾值設定策略、以及如何與服務降級配合，缺乏深入理解。本文將以 Resilience4j 為例，帶你實作一套完整的熔斷與降級機制。

## 熔斷器的三態模型解析

熔斷器的運作基於三種狀態的轉換：**Closed**（關閉）、**Open**（開啟）、**Half-Open**（半開）。

```
┌─────────┐  失敗率超過閾值  ┌─────────┐
│  Closed │ ──────────────→ │  Open   │
└─────────┘                  └─────────┘
     ↑                            │
     │     成功率達標              │ 等待時間結束
     │  ┌─────────────┐           │
     └──│  Half-Open  │←──────────┘
        └─────────────┘
              │
              │ 失敗率仍高
              ↓
        回到 Open 狀態
```

- **Closed**：正常狀態，請求直接通過，同時記錄成功/失敗次數
- **Open**：熔斷狀態，所有請求直接失敗，執行降級邏輯
- **Half-Open**：試探狀態，允許有限請求通過，測試下游是否恢復

理解這三態的轉換條件，是正確配置熔斷器的基礎。

## Resilience4j 核心配置實作

以下是一個 Spring Boot 整合 Resilience4j 的實際配置範例：

```yaml
# application.yml
resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        # 使用滑動視窗統計，基於最近 10 次呼叫
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 10
        # 失敗率達到 50% 時開啟熔斷
        failureRateThreshold: 50
        # 慢呼叫（超過 2 秒）比例達 80% 也觸發熔斷
        slowCallRateThreshold: 80
        slowCallDurationThreshold: 2s
        # Open 狀態持續 30 秒後進入 Half-Open
        waitDurationInOpenState: 30s
        # Half-Open 狀態允許 5 個試探請求
        permittedNumberOfCallsInHalfOpenState: 5
        # 記錄這些例外為失敗
        recordExceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
          - feign.FeignException
```

這裡有幾個關鍵參數需要根據業務特性調整：

- `slidingWindowSize`：視窗太小會過於敏感，太大則反應遲鈍
- `waitDurationInOpenState`：需考量下游服務的平均恢復時間
- `slowCallDurationThreshold`：應略高於正常 P99 響應時間

## 服務降級的 Fallback 策略

熔斷器開啟後，必須搭配合理的降級策略，否則使用者只會看到一堆錯誤：

```java
@Service
public class PaymentService {

    @CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
    @TimeLimiter(name = "paymentService")
    public CompletableFuture<PaymentResult> processPayment(PaymentRequest request) {
        return CompletableFuture.supplyAsync(() -> 
            paymentClient.process(request)
        );
    }

    // 降級方法：參數需與原方法一致，最後加上 Throwable
    private CompletableFuture<PaymentResult> paymentFallback(
            PaymentRequest request, Throwable t) {
        
        log.warn("Payment service degraded, reason: {}", t.getMessage());
        
        // 策略一：返回快取的預設結果
        if (t instanceof CallNotPermittedException) {
            return CompletableFuture.completedFuture(
                PaymentResult.pending("系統繁忙，訂單已暫存，稍後處理")
            );
        }
        
        // 策略二：寫入訊息佇列，非同步重試
        messageQueue.send("payment-retry-queue", request);
        return CompletableFuture.completedFuture(
            PaymentResult.queued("訂單已排入處理佇列")
        );
    }
}
```

良好的降級策略應該讓使用者感知到「服務仍在運作」，而非直接報錯。

## 監控與告警整合

熔斷器的狀態變化必須被監控，以下是整合 Prometheus 與 Grafana 的作法：

```java
@Configuration
public class CircuitBreakerMetricsConfig {

    @Bean
    public CircuitBreakerRegistry circuitBreakerRegistry(
            MeterRegistry meterRegistry) {
        
        CircuitBreakerRegistry registry = CircuitBreakerRegistry.ofDefaults();
        
        // 註冊狀態變化事件
        registry.getEventPublisher()
            .onStateTransition(event -> {
                String name = event.getCircuitBreakerName();
                String from = event.getStateTransition().getFromState().name();
                String to = event.getStateTransition().getToState().name();
                
                log.warn("CircuitBreaker [{}] 狀態變化: {} -> {}", name, from, to);
                
                // 狀態變為 OPEN 時發送告警
                if ("OPEN".equals(to)) {
                    alertService.sendSlackAlert(
                        String.format("🚨 熔斷器 %s 已開啟！", name)
                    );
                }
            });
        
        // 綁定 Micrometer 指標
        TaggedCircuitBreakerMetrics.ofCircuitBreakerRegistry(registry)
            .bindTo(meterRegistry);
        
        return registry;
    }
}
```

在 Grafana 中可設定以下告警規則：

```promql
# 當熔斷器處於 OPEN 狀態超過 1 分鐘時告警
resilience4j_circuitbreaker_state{state="open"} == 1
```
