# 實戰解析：Resilience4j 熔斷器的三種狀態與最佳閾值配置

## 前言

當微服務架構成為主流，服務之間的依賴關係也越來越複雜。一個下游服務的延遲或故障，很容易引發連鎖反應，導致整個系統雪崩。熔斷器（Circuit Breaker）正是解決這個問題的關鍵模式，而「如何正確配置熔斷器的閾值」往往是中階工程師在實作時最容易踩坑的地方。

本文將以 Java 生態系中最主流的 Resilience4j 為例，深入探討熔斷器的三種狀態轉換機制，並提供實戰中驗證過的閾值配置建議，幫助你避免「熔斷太敏感導致誤殺」或「熔斷太遲鈍形同虛設」的兩難困境。

## 熔斷器的三種狀態與轉換邏輯

Resilience4j 的熔斷器基於有限狀態機運作，包含三種核心狀態：

- **CLOSED（關閉）**：正常狀態，所有請求都會通過並被監控
- **OPEN（開啟）**：熔斷狀態，所有請求直接被拒絕，快速失敗
- **HALF_OPEN（半開）**：試探狀態，允許部分請求通過以測試下游是否恢復

狀態轉換的觸發條件由你配置的閾值決定：

```
CLOSED ---(失敗率超過閾值)---> OPEN
OPEN ---(等待時間結束)---> HALF_OPEN  
HALF_OPEN ---(試探成功)---> CLOSED
HALF_OPEN ---(試探失敗)---> OPEN
```

理解這個狀態機是正確配置的基礎，接下來我們看具體參數。

## 核心閾值參數詳解

以下是 Resilience4j CircuitBreaker 最關鍵的配置參數：

```yaml
resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        # 滑動窗口類型：COUNT_BASED 或 TIME_BASED
        slidingWindowType: COUNT_BASED
        # 滑動窗口大小
        slidingWindowSize: 100
        # 觸發熔斷的失敗率閾值（百分比）
        failureRateThreshold: 50
        # 觸發熔斷的慢呼叫率閾值（百分比）
        slowCallRateThreshold: 80
        # 定義慢呼叫的時間門檻
        slowCallDurationThreshold: 3s
        # OPEN 狀態持續時間
        waitDurationInOpenState: 30s
        # HALF_OPEN 狀態允許的試探請求數
        permittedNumberOfCallsInHalfOpenState: 10
        # 最小呼叫數量（未達此數不計算失敗率）
        minimumNumberOfCalls: 20
```

這裡有個常見錯誤：將 `minimumNumberOfCalls` 設得太低。如果設為 5，那麼 5 個請求中只要 3 個失敗就會觸發熔斷，這在流量低峰時段很容易造成誤判。

## 根據服務特性調整配置策略

不同類型的服務需要不同的熔斷策略，以下是三種典型場景：

**高頻核心服務（如訂單查詢）**：
```java
CircuitBreakerConfig.custom()
    .slidingWindowType(SlidingWindowType.TIME_BASED)
    .slidingWindowSize(10) // 10秒的時間窗口
    .failureRateThreshold(30) // 較低閾值，快速反應
    .waitDurationInOpenState(Duration.ofSeconds(15))
    .build();
```

**低頻批次服務（如報表產生）**：
```java
CircuitBreakerConfig.custom()
    .slidingWindowType(SlidingWindowType.COUNT_BASED)
    .slidingWindowSize(20)
    .failureRateThreshold(60) // 較高容忍度
    .minimumNumberOfCalls(10)
    .waitDurationInOpenState(Duration.ofSeconds(60))
    .build();
```

**第三方 API 整合**：
```java
CircuitBreakerConfig.custom()
    .slowCallRateThreshold(70) // 重視延遲監控
    .slowCallDurationThreshold(Duration.ofSeconds(5))
    .waitDurationInOpenState(Duration.ofSeconds(45))
    .build();
```

## 搭配服務降級實現優雅失敗

熔斷器觸發後，必須搭配適當的 fallback 機制才能提供良好的使用者體驗：

```java
@Service
public class ProductService {
    
    private final CircuitBreaker circuitBreaker;
    private final ProductClient productClient;
    private final CacheManager cacheManager;
    
    public Product getProduct(String productId) {
        Supplier<Product> decoratedSupplier = CircuitBreaker
            .decorateSupplier(circuitBreaker, 
                () -> productClient.fetchProduct(productId));
        
        return Try.ofSupplier(decoratedSupplier)
            .recover(CallNotPermittedException.class, 
                e -> getFallbackProduct(productId))  // 熔斷時走降級
            .recover(Exception.class, 
                e -> getCachedProduct(productId))    // 其他異常走快取
            .get();
    }
    
    private Product getFallbackProduct(String productId) {
        // 返回快取資料或預設值
        return cacheManager.getCache("products")
            .get(productId, Product.class);
    }
}
```

## 監控與動態調整

配置不是一次性的工作，你需要持續監控並調整。Resilience4j 提供了完整的 metrics 整合：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,circuitbreakers,metrics
  health:
    circuitbreakers:
      enabled: true
```

透過 Prometheus + Grafana 建立儀表板，重點監控以下指標：

- `resilience4j_circuitbreaker_state`：當前狀態
- `resilience4j_circuitbreaker_failure_rate`：實際失敗率
- `resilience4j_circuitbreaker_not_permitted_calls_total`：被熔斷的請求數

## 結論與行動建議

熔斷器的閾值配置沒有標準答案，但有幾個原則值得遵循：

1. **從保守開始**：先設定較高的失敗率閾值（如 60%），觀察一週後再逐步調低
2. **區分服務等級**：核心服務用較敏感的配置，邊緣服務可以寬鬆些
3. **務必設定 minimumNumberOfCalls**：避免低流量時的統計偏差
4. **fallback 不能省**：熔斷
