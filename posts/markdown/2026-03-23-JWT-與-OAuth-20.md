# 解決 JWT Token 過期困境：OAuth 2.0 Refresh Token 輪替策略實戰指南

## 前言

在實作 OAuth 2.0 認證流程時，許多團隊都會遇到一個棘手問題：JWT Access Token 設定太短，使用者頻繁被登出體驗差；設定太長，又增加 Token 被盜用的風險窗口。這個看似兩難的問題，其實可以透過正確的 Refresh Token 輪替（Rotation）策略來優雅解決。

本文將深入探討如何在 OAuth 2.0 架構下，實作安全且使用者友善的 Token 更新機制，並提供可直接套用的實戰配置。

## Access Token 與 Refresh Token 的角色分工

在 OAuth 2.0 規範中，這兩種 Token 各司其職：

- **Access Token**：短效期（通常 15-60 分鐘），用於 API 請求授權
- **Refresh Token**：長效期（數天至數週），僅用於取得新的 Access Token

正確的策略是讓 Access Token 保持短效期以限制風險，同時透過 Refresh Token 實現無感更新。以下是典型的 JWT Access Token payload 結構：

```json
{
  "sub": "user_12345",
  "iss": "https://auth.example.com",
  "aud": "https://api.example.com",
  "exp": 1711180800,
  "iat": 1711177200,
  "scope": "read:profile write:data"
}
```

關鍵在於 `exp`（過期時間）與 `iat`（簽發時間）的差距，建議控制在 15-30 分鐘內。

## 實作 Refresh Token 輪替機制

Refresh Token Rotation 的核心概念是：每次使用 Refresh Token 換取新的 Access Token 時，同時發放新的 Refresh Token 並廢棄舊的。這能有效降低 Refresh Token 被盜用的風險。

以 Node.js 搭配常見的認證服務為例，後端處理邏輯如下：

```javascript
// token-refresh.js
const refreshTokens = new Map(); // 實務上應使用 Redis

async function handleTokenRefresh(oldRefreshToken) {
  // 1. 驗證 Refresh Token 是否有效且未被使用
  const tokenData = refreshTokens.get(oldRefreshToken);
  if (!tokenData || tokenData.used) {
    // 偵測到重複使用，可能是 Token 被盜
    await revokeAllUserTokens(tokenData?.userId);
    throw new Error('REFRESH_TOKEN_REUSE_DETECTED');
  }

  // 2. 標記舊 Token 為已使用
  tokenData.used = true;
  refreshTokens.set(oldRefreshToken, tokenData);

  // 3. 產生新的 Token 組合
  const newAccessToken = generateJWT({ 
    sub: tokenData.userId,
    exp: Math.floor(Date.now() / 1000) + 900 // 15 分鐘
  });
  
  const newRefreshToken = generateSecureToken();
  refreshTokens.set(newRefreshToken, {
    userId: tokenData.userId,
    used: false,
    createdAt: Date.now()
  });

  return { accessToken: newAccessToken, refreshToken: newRefreshToken };
}
```

## 前端配合：實作 Silent Refresh

後端機制到位後，前端需要配合實作無感更新。使用 Axios 攔截器是常見做法：

```typescript
// api-client.ts
import axios from 'axios';

const apiClient = axios.create({
  baseURL: 'https://api.example.com'
});

let isRefreshing = false;
let failedQueue: Array<{ resolve: Function; reject: Function }> = [];

apiClient.interceptors.response.use(
  (response) => response,
  async (error) => {
    const originalRequest = error.config;
    
    if (error.response?.status === 401 && !originalRequest._retry) {
      if (isRefreshing) {
        // 等待其他請求完成 refresh
        return new Promise((resolve, reject) => {
          failedQueue.push({ resolve, reject });
        }).then(() => apiClient(originalRequest));
      }

      originalRequest._retry = true;
      isRefreshing = true;

      try {
        const { accessToken, refreshToken } = await refreshTokenAPI();
        saveTokens(accessToken, refreshToken);
        failedQueue.forEach(({ resolve }) => resolve());
        return apiClient(originalRequest);
      } finally {
        isRefreshing = false;
        failedQueue = [];
      }
    }
    return Promise.reject(error);
  }
);
```

## 安全性強化：搭配 Token Binding 與監控

除了輪替機制，建議加入以下安全措施：

```yaml
# auth-service-config.yaml
token:
  access:
    ttl: 900          # 15 分鐘
    algorithm: RS256  # 使用非對稱加密
  refresh:
    ttl: 604800       # 7 天
    rotation: true
    reuse_detection: true
    max_age_absolute: 2592000  # 30 天強制重新登入

security:
  bind_to_ip: false   # Mobile 環境不建議
  bind_to_fingerprint: true
  rate_limit:
    refresh_per_hour: 10
```

同時，務必監控異常的 Token 更新模式：

```bash
# 使用 Prometheus 監控可疑的 refresh 行為
rate(auth_refresh_token_reuse_total[5m]) > 0
rate(auth_refresh_requests_total[1h]) > 50
```

## 結論與行動建議

正確實作 Refresh Token 輪替策略，能在不犧牲使用者體驗的前提下，大幅提升系統安全性。建議的行動步驟：

1. **審視現有配置**：確認 Access Token TTL 不超過 30 分鐘
2. **啟用輪替機制**：每次 refresh 都發放新的 Refresh Token
3. **實作重複使用偵測**：發現異常立即撤銷該使用者所有 Token
4. **部署監控告警**：追蹤異常的 Token 更新頻率
5. **定期強制重新認證**：設定 Refresh Token 的絕對有效期限

這些措施能有效平衡安全性與使用體驗，是每個處理認證系統的 DevOps 工程師必備的實戰技能。
