# Docker 多階段建構：讓你的映像從肥胖變精實的實戰指南

## 前言

你是否曾經建構出一個超過 1GB 的 Docker 映像，卻只是為了跑一個簡單的 Node.js 或 Go 應用程式？這不僅浪費儲存空間，更會拖慢 CI/CD Pipeline 的執行速度、增加網路傳輸成本，甚至擴大潛在的安全攻擊面。

多階段建構（Multi-stage Build）是 Docker 17.05 版本引入的功能，它讓我們能在單一 Dockerfile 中使用多個 `FROM` 指令，將建構環境與執行環境完美分離。這意味著最終映像只包含執行應用程式所需的最小檔案集合，而非整個編譯工具鏈。對於追求精實部署的 DevOps 團隊而言，這是不可或缺的技能。

## 理解多階段建構的核心概念

傳統的 Dockerfile 通常將所有步驟塞在單一階段中，導致編譯器、開發依賴、暫存檔案全部殘留在最終映像裡。多階段建構的精髓在於：**每個階段都是獨立的建構環境，而你可以選擇性地從前一階段複製所需的產出物**。

以下是基本的多階段建構結構：

```dockerfile
# 階段一：建構階段
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build

# 階段二：執行階段
FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 3000
CMD ["node", "dist/main.js"]
```

關鍵語法是 `COPY --from=builder`，它讓我們能從名為 `builder` 的階段中擷取特定檔案，而不攜帶該階段的其他內容。

## 實戰案例：Go 應用程式的極致瘦身

Go 語言因為能編譯成靜態連結的二進位檔，特別適合展示多階段建構的威力。讓我們比較傳統做法與多階段建構的差異：

```dockerfile
# 傳統做法 - 最終映像約 800MB
FROM golang:1.22
WORKDIR /app
COPY . .
RUN go build -o main .
CMD ["./main"]
```

```dockerfile
# 多階段建構 - 最終映像約 12MB
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o main .

FROM scratch
COPY --from=builder /app/main /main
ENTRYPOINT ["/main"]
```

這裡有幾個優化要點：
- `CGO_ENABLED=0` 確保產生純靜態二進位檔
- `-ldflags="-s -w"` 移除除錯符號，進一步縮小檔案
- 使用 `scratch` 作為基礎映像，這是一個完全空白的映像

透過這些技巧，我們將映像大小從 800MB 降到 12MB，縮減了 98%！

## 進階技巧：善用快取與並行建構

在複雜的專案中，我們可以運用更多技巧來優化建構流程：

### 利用獨立階段進行並行建構

```dockerfile
FROM node:20-alpine AS frontend-builder
WORKDIR /frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ .
RUN npm run build

FROM golang:1.22-alpine AS backend-builder
WORKDIR /backend
COPY backend/go.mod backend/go.sum ./
RUN go mod download
COPY backend/ .
RUN go build -o server .

FROM alpine:3.19 AS runner
RUN apk --no-cache add ca-certificates
WORKDIR /app
COPY --from=backend-builder /backend/server .
COPY --from=frontend-builder /frontend/dist ./static
EXPOSE 8080
CMD ["./server"]
```

當使用 BuildKit（`DOCKER_BUILDKIT=1`）時，Docker 會自動並行執行沒有相依關係的階段，大幅加速建構時間。

### 選擇適當的基礎映像

| 基礎映像 | 大小 | 適用場景 |
|---------|------|---------|
| `scratch` | 0MB | 靜態編譯的 Go/Rust 應用 |
| `alpine` | ~5MB | 需要基本 shell 和套件管理 |
| `distroless` | ~20MB | 需要 glibc 但不需 shell |
| `slim` 變體 | ~50-80MB | 需要較完整的執行環境 |

Google 的 Distroless 映像是個優秀的折衷方案：

```dockerfile
FROM gcr.io/distroless/static-debian12
COPY --from=builder /app/main /
CMD ["/main"]
```

## 除錯與驗證你的成果

建構完成後，務必驗證映像大小和內容：

```bash
# 檢視映像大小
docker images your-app --format "{{.Repository}}:{{.Tag}} - {{.Size}}"

# 分析映像層級
docker history your-app:latest

# 深度分析工具
dive your-app:latest
```

`dive` 是一個優秀的開源工具，能視覺化呈現每一層的檔案變化，幫助你找出不必要的檔案。

另外，建議在 CI/CD Pipeline 中加入映像大小檢查：

```yaml
# GitLab CI 範例
check_image_size:
  script:
    - SIZE=$(docker inspect your-app:latest --format='{{.Size}}')
    - |
      if [ $SIZE -gt 100000000 ]; then
        echo "映像大小超過 100MB 限制！"
        exit 1
      fi
```

## 結論與行動建議

多階段建構是現代容器化部署的必備技能。透過本文介紹的技巧，你應該能夠：

1. **立即行動**：檢視現有專案的 Dockerfile，識別可以分離的建構與執行階段
2. **選擇正確的基礎映像**：根據應用需求選擇最小可行的映像（scratch > alpine > distroless > slim）
3. **建立品質關卡**：在 CI/CD 中加入映像大小監控，防止映像肥大化
4. **啟用 BuildKit**：設定 `DOCKER_BUILDKIT=1` 獲得更好的快取機制和並行建構能力

記住，每減少 100MB 的映像大小，在大規模部署時都會轉化為可觀的時間和成本節省。從今天開始重構你的 Dockerfile 吧！
