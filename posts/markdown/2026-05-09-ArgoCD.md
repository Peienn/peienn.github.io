# ArgoCD Application 同步卡住？深入解析 Sync Stuck 問題與排除技巧

## 前言

在使用 ArgoCD 進行 GitOps 部署時，你是否曾遇過 Application 狀態一直顯示「Syncing」或「Progressing」，遲遲無法完成同步？這個問題在生產環境中特別令人頭痛，因為它可能導致部署流程停滯，進而影響整體 CI/CD pipeline。

Sync Stuck 問題的成因多元，從簡單的資源配置錯誤到複雜的 Kubernetes Controller 問題都有可能。本文將從實務角度出發，分享常見的卡住情境與對應的排除技巧，幫助你快速定位問題根源並恢復部署流程。

## 快速診斷：確認 Application 當前狀態

當發現同步卡住時，第一步是釐清 Application 目前的 Sync 與 Health 狀態。使用 CLI 可以快速取得詳細資訊：

```bash
# 查看 Application 整體狀態
argocd app get <app-name> --refresh

# 檢視詳細的資源同步狀態
argocd app resources <app-name>

# 取得 Application 的 YAML 完整輸出，包含 status 區塊
kubectl get application <app-name> -n argocd -o yaml
```

重點觀察 `status.operationState` 與 `status.resources` 欄位。若看到特定資源的 `health.status` 為 `Progressing` 或 `Degraded`，就能縮小問題範圍。

## 常見情境一：Resource Hook 執行逾時

ArgoCD 的 Resource Hook（如 PreSync、PostSync Job）若執行時間過長或失敗，會導致整個 Sync 流程卡住。預設情況下，ArgoCD 會無限期等待 Hook 完成。

**排除步驟：**

```bash
# 列出目前正在執行的 Hook
kubectl get jobs -n <namespace> -l argocd.argoproj.io/hook

# 檢查 Job 的執行狀態與 Pod logs
kubectl logs job/<hook-job-name> -n <namespace>
```

**解決方案：** 在 Hook 資源加入逾時設定：

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
  annotations:
    argocd.argoproj.io/hook: PreSync
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
    argocd.argoproj.io/sync-wave: "-1"
spec:
  activeDeadlineSeconds: 300  # 設定 5 分鐘逾時
  template:
    spec:
      containers:
      - name: migrate
        image: myapp/migration:latest
      restartPolicy: Never
  backoffLimit: 1
```

## 常見情境二：Finalizer 阻擋資源刪除

當 Application 嘗試刪除舊資源，但該資源的 Finalizer 無法正常移除時，Sync 會永遠停在刪除階段。這常見於 CRD 資源或使用自訂 Controller 的情境。

**診斷方式：**

```bash
# 找出卡住的資源
kubectl get <resource-type> -n <namespace> -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.metadata.finalizers}{"\n"}{end}'

# 檢查資源是否有 deletionTimestamp 但仍存在
kubectl get <resource-type> <name> -n <namespace> -o yaml | grep -A5 metadata
```

**緊急處理（謹慎使用）：**

```bash
# 手動移除 Finalizer（確認不會造成資料不一致後才執行）
kubectl patch <resource-type> <name> -n <namespace> \
  --type='json' \
  -p='[{"op": "remove", "path": "/metadata/finalizers"}]'
```

## 常見情境三：Health Check 自訂規則配置錯誤

ArgoCD 使用 Lua 腳本判斷資源健康狀態。若使用自訂 CRD 且未正確配置 Health Check，ArgoCD 可能永遠判定資源為 `Progressing`。

**在 argocd-cm ConfigMap 中加入自訂規則：**

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-cm
  namespace: argocd
data:
  resource.customizations.health.mycrd.example.com_MyResource: |
    hs = {}
    if obj.status ~= nil then
      if obj.status.phase == "Ready" then
        hs.status = "Healthy"
        hs.message = "Resource is ready"
      elseif obj.status.phase == "Failed" then
        hs.status = "Degraded"
        hs.message = obj.status.message or "Resource failed"
      else
        hs.status = "Progressing"
        hs.message = "Resource is being processed"
      end
    else
      hs.status = "Progressing"
      hs.message = "Waiting for status"
    end
    return hs
```

修改後記得重啟 argocd-repo-server 以載入新規則：

```bash
kubectl rollout restart deployment argocd-repo-server -n argocd
```

## 預防措施：設定 Sync 操作逾時

為避免 Sync 無限期等待，建議在 Application 層級設定操作逾時：

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
spec:
  syncPolicy:
    retry:
      limit: 3
      backoff:
        duration: 5s
        maxDuration: 3m
        factor: 2
  # 在 Operation 層級可透過 CLI 指定逾時
```

執行同步時指定逾時：

```bash
argocd app sync <app-name> --timeout 600
```

## 結論與行動建議

ArgoCD Sync Stuck 問題雖然惱人，但透過系統化的診斷流程，多數情況都能快速定位根因。建議團隊採取以下行動：

1. **建立標準 Runbook**：將本文的診斷指令整理成團隊 SOP
2. **為所有 Hook Job 設定逾時**：避免單一 Job 拖垮整條部署流程
3. **自訂 CRD 務必配置 Health Check**：確保 ArgoCD 能正確判斷資源狀態
4. **啟用 Notification**：搭配 argocd-notifications 在 Sync 異常時即時告警

掌握這些技巧後，你將能更從容地應對 GitOps 部署中的各種挑戰。
