# 資料庫連線池調優：別讓連線耗盡成為你的凌晨三點噩夢

## 前言

「資料庫連線池已耗盡，服務無法響應。」這則告警訊息，相信許多 DevOps 工程師都曾在深夜收到過。連線池耗盡（Connection Pool Exhaustion）是生產環境中最常見卻也最容易被忽視的效能瓶頸之一。

當應用程式流量突然飆升，或是某段程式碼忘記釋放連線，連線池很快就會被掏空。結果就是新的請求無法獲取連線，服務開始排隊、逾時，最終導致整體系統癱瘓。本文將從實戰角度出發，分享連線池調優的關鍵策略，幫助你建立更穩健的資料庫連線管理機制。

## 理解連線池的核心參數

在開始調優之前，必須先掌握連線池的基本參數。以 HikariCP（Java 生態系中最受歡迎的連線池）為例，以下是最關鍵的設定：

```yaml
# HikariCP 關鍵配置範例
spring:
  datasource:
    hikari:
      maximum-pool-size: 20        # 最大連線數
      minimum-idle: 5              # 最小閒置連線數
      connection-timeout: 30000    # 獲取連線的等待時間（毫秒）
      idle-timeout: 600000         # 閒置連線存活時間
      max-lifetime: 1800000        # 連線最大存活時間
      leak-detection-threshold: 60000  # 連線洩漏偵測閾值
```

`maximum-pool-size` 並非越大越好。根據 HikariCP 官方建議，合理的連線數公式為：`connections = (CPU 核心數 * 2) + 有效磁碟數`。對於大多數 OLTP 場景，10-20 個連線已足以處理相當高的並發量。

## 監控指標：讓問題無所遁形

沒有監控就沒有調優。建立完善的連線池監控是預防問題的第一道防線。以下是使用 Prometheus 搭配 Grafana 監控 HikariCP 的實作方式：

```java
// 暴露 HikariCP 指標給 Prometheus
@Configuration
public class MetricsConfig {
    @Bean
    public MeterRegistryCustomizer<MeterRegistry> metricsCommonTags() {
        return registry -> registry.config().commonTags("application", "my-service");
    }
}
```

關鍵監控指標包括：

```promql
# 活躍連線數
hikaricp_connections_active{pool="HikariPool-1"}

# 等待連線的執行緒數（重要！）
hikaricp_connections_pending{pool="HikariPool-1"}

# 連線獲取時間（p99）
histogram_quantile(0.99, hikaricp_connections_acquire_seconds_bucket)
```

當 `connections_pending` 持續大於 0，或連線獲取時間明顯上升，就是連線池即將耗盡的前兆。建議設定告警規則，在問題惡化前主動介入：

```yaml
# Prometheus 告警規則
- alert: ConnectionPoolNearExhaustion
  expr: hikaricp_connections_pending > 5
  for: 2m
  labels:
    severity: warning
  annotations:
    summary: "連線池接近耗盡，待處理請求數過高"
```

## 連線洩漏偵測與根因分析

連線洩漏是造成連線池耗盡的常見元兇。當程式碼未正確關閉連線時，連線會被永久佔用。啟用洩漏偵測機制可以快速定位問題：

```properties
# 設定 60 秒未歸還連線即視為洩漏
hikari.leak-detection-threshold=60000
```

啟用後，當偵測到潛在洩漏，日誌會印出完整的 stack trace：

```
WARN  com.zaxxer.hikari.pool.ProxyLeakTask - Connection leak detection triggered for connection org.postgresql.jdbc.PgConnection@1a2b3c4d, stack trace follows
java.lang.Exception: Apparent connection leak detected
    at com.example.service.UserService.findUserById(UserService.java:45)
    at com.example.controller.UserController.getUser(UserController.java:28)
```

有了 stack trace，就能精確定位到哪段程式碼忘記關閉連線。現代框架通常推薦使用 try-with-resources 語法確保連線釋放：

```java
// 正確的連線管理方式
try (Connection conn = dataSource.getConnection();
     PreparedStatement stmt = conn.prepareStatement(sql)) {
    // 執行查詢
} // 連線會自動歸還給連線池
```

## 動態調整與彈性擴展策略

在雲原生環境中，固定的連線池設定往往無法應對流量波動。可以透過以下策略實現彈性調整：

**方法一：基於 ConfigMap 的動態配置（Kubernetes）**

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: db-pool-config
data:
  HIKARI_MAX_POOL_SIZE: "30"
  HIKARI_MIN_IDLE: "10"
```

**方法二：使用 PgBouncer 作為連線池代理**

當多個服務實例連接同一資料庫時，每個實例都維護獨立連線池會導致連線數爆炸。PgBouncer 可以在資料庫前端統一管理連線：

```ini
# pgbouncer.ini
[databases]
mydb = host=postgres port=5432 dbname=mydb

[pgbouncer]
pool_mode = transaction    # 交易級別的連線共享
max_client_conn = 1000     # 最大客戶端連線數
default_pool_size = 50     # 每個資料庫的預設連線池大小
reserve_pool_size = 10     # 保留連線數應對突發流量
```

使用 transaction 模式可以大幅提升連線利用率，因為連線只在交易執行期間被佔用。

## 結論與行動建議

資料庫連線池調優不是一次性工作，而是持續的觀察與調整過程。以下是立即可執行的行動清單：

1. **盤點現況**：檢視目前各服務的連線池設定，確認是否符合應用特性
2. **建立監控**：部署連線池指標收集，設定 pending connections 和獲取時間的告警
3. **啟用洩漏偵測**：將 `leak-detection-threshold` 設為合理值（建議 30-60 秒）
4. **定期壓測**：模擬高並發場景，驗證連線池在極端情況下的表現
