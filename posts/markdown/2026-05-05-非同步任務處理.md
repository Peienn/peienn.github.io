# 從 Celery 到 Redis Streams：重新思考非同步任務的重試策略

## 前言

在分散式系統中，非同步任務失敗是常態而非例外。網路抖動、下游服務過載、資源競爭——這些都可能導致任務執行失敗。作為 DevOps 工程師，我們經常面對的問題不是「任務會不會失敗」，而是「任務失敗後該如何優雅地重試」。

設計不良的重試策略會帶來災難性後果：指數級的請求放大效應可能壓垮下游服務，無限重試會耗盡系統資源，而過於保守的策略則會導致任務無謂地被丟棄。本文將探討如何在 Celery 與 Redis Streams 兩種常見方案中，實作一套兼顧可靠性與系統穩定性的重試機制。

## 理解指數退避與抖動的必要性

最基本的重試策略是固定間隔重試，但這在分散式環境中會造成「驚群效應」（Thundering Herd）。當大量任務同時失敗並在相同時間點重試，會對系統造成瞬間高峰。

指數退避（Exponential Backoff）搭配隨機抖動（Jitter）是標準解法：

```python
import random

def calculate_backoff(attempt: int, base_delay: float = 1.0, max_delay: float = 300.0) -> float:
    """計算帶有完全抖動的指數退避時間"""
    exponential_delay = base_delay * (2 ** attempt)
    capped_delay = min(exponential_delay, max_delay)
    # 完全抖動：在 0 到 capped_delay 之間隨機選擇
    return random.uniform(0, capped_delay)
```

這個公式確保重試請求在時間軸上均勻分散，避免對下游服務造成週期性壓力。

## Celery 的進階重試配置

Celery 內建的重試機制已經相當成熟，但預設配置往往過於寬鬆。以下是一個生產環境推薦的配置模式：

```python
from celery import Celery
from celery.exceptions import MaxRetriesExceededError

app = Celery('tasks')

@app.task(
    bind=True,
    max_retries=5,
    default_retry_delay=60,
    autoretry_for=(ConnectionError, TimeoutError),
    retry_backoff=True,
    retry_backoff_max=600,
    retry_jitter=True,
    acks_late=True,
    reject_on_worker_lost=True
)
def process_payment(self, order_id: str):
    try:
        result = call_payment_gateway(order_id)
        return result
    except PaymentGatewayError as exc:
        # 針對特定錯誤類型使用不同的重試策略
        raise self.retry(
            exc=exc,
            countdown=calculate_backoff(self.request.retries),
            max_retries=3  # 覆蓋預設值
        )
```

關鍵配置說明：
- `acks_late=True`：任務完成後才確認，防止 Worker 意外終止導致任務遺失
- `reject_on_worker_lost=True`：Worker 遺失時將任務重新入隊
- `retry_jitter=True`：啟用內建的抖動機制

## Redis Streams 的死信佇列模式

當 Celery 的複雜度超出需求，或需要更細粒度的控制時，Redis Streams 提供了輕量級的替代方案。其 Consumer Group 機制天然支援任務確認與 Pending 狀態追蹤：

```python
import redis
import json
from datetime import datetime, timedelta

r = redis.Redis(host='localhost', port=6379, decode_responses=True)

STREAM_NAME = 'tasks:main'
DLQ_STREAM = 'tasks:dlq'
GROUP_NAME = 'workers'
MAX_RETRIES = 3

def claim_and_retry_stale_tasks(idle_time_ms: int = 60000):
    """認領並重試超時的 Pending 任務"""
    pending = r.xpending_range(
        STREAM_NAME, GROUP_NAME, 
        min='-', max='+', count=100
    )
    
    for entry in pending:
        message_id = entry['message_id']
        delivery_count = entry['times_delivered']
        
        if delivery_count > MAX_RETRIES:
            # 移至死信佇列
            messages = r.xrange(STREAM_NAME, min=message_id, max=message_id)
            if messages:
                r.xadd(DLQ_STREAM, {
                    'original_id': message_id,
                    'payload': json.dumps(messages[0][1]),
                    'failed_at': datetime.utcnow().isoformat()
                })
            r.xack(STREAM_NAME, GROUP_NAME, message_id)
        elif entry['time_since_delivered'] > idle_time_ms:
            # 重新認領任務
            r.xclaim(
                STREAM_NAME, GROUP_NAME, 'retry-worker',
                min_idle_time=idle_time_ms,
                message_ids=[message_id]
            )
```

這種模式的優勢在於：重試次數由 Redis 自動追蹤（`times_delivered`），且 Pending 狀態可視化，便於監控與除錯。

## 監控指標與告警閾值

無論採用哪種方案，完善的可觀測性都是重試策略成功的關鍵。以下是必須追蹤的核心指標：

```yaml
# Prometheus 告警規則範例
groups:
  - name: async_task_alerts
    rules:
      - alert: HighTaskRetryRate
        expr: |
          rate(celery_task_retries_total[5m]) / 
          rate(celery_task_attempts_total[5m]) > 0.1
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "任務重試率超過 10%"
          
      - alert: DLQBacklogGrowing
        expr: redis_stream_length{stream="tasks:dlq"} > 100
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "死信佇列積壓超過 100 筆"
```

建議設定三層告警：重試率異常（預警）、死信佇列積壓（嚴重）、任務處理延遲 P99 超標（影響用戶）。

## 結論與行動建議

設計非同步任務的重試策略，核心原則是「對自己嚴格，對下游友善」。具體行動建議：

1. **立即執行**：檢視現有任務的重試配置，確認是否啟用指數退避與抖動
2. **短期優化**：為所有任務佇列建立死信佇列，避免問題任務無限循環
