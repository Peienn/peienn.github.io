# 用 Celery + Redis 打造可觀測的非同步任務系統：從 Silent Failure 到全面監控

## 前言

在微服務架構中，非同步任務處理是不可或缺的一環。無論是寄送 Email、處理圖片、產生報表，我們都依賴 Celery 這類任務佇列來處理耗時操作。然而，許多團隊在導入 Celery 後，常陷入一個困境：**任務失敗了，但沒人知道**。

這種「Silent Failure」問題在生產環境中尤其危險。用戶抱怨沒收到通知信、報表遲遲沒產出，工程師卻在茫茫日誌中找不到線索。本文將分享如何為 Celery + Redis 的任務系統建立完整的可觀測性，讓每一個任務的狀態都無所遁形。

## 問題診斷：為什麼你的任務會「靜默失敗」

在深入解法前，先釐清常見的 Silent Failure 來源：

1. **Worker 記憶體爆掉被 OOM Killer 終結** — 任務直接消失
2. **任務超時但沒設 `soft_time_limit`** — 卡死佔用 Worker
3. **序列化失敗** — 任務進不了佇列
4. **Broker 連線中斷** — 任務丟失

檢查你目前的設定是否存在這些風險：

```python
# celery_config.py - 常見的危險配置
app = Celery('myapp')

# ❌ 沒有設定任務結果後端，無法追蹤狀態
app.conf.result_backend = None

# ❌ 沒有時間限制，任務可能無限卡死
app.conf.task_time_limit = None
```

## 基礎建設：Result Backend 與任務追蹤

第一步是確保每個任務的執行結果都被記錄。使用 Redis 作為 Result Backend 是常見選擇：

```python
# celery_config.py - 可觀測性基礎配置
from celery import Celery

app = Celery('myapp')

app.conf.update(
    broker_url='redis://localhost:6379/0',
    result_backend='redis://localhost:6379/1',
    
    # 結果保留 24 小時，方便事後查詢
    result_expires=86400,
    
    # 追蹤任務開始狀態，不只是完成
    task_track_started=True,
    
    # 時間限制：soft limit 發 signal，hard limit 強制終止
    task_soft_time_limit=300,
    task_time_limit=360,
    
    # 任務失敗時保留完整 traceback
    task_acks_late=True,
    task_reject_on_worker_lost=True,
)
```

透過 `task_track_started=True`，你可以在 Redis 中看到任務從 PENDING → STARTED → SUCCESS/FAILURE 的完整生命週期。

## 信號機制：捕捉每一個關鍵時刻

Celery 提供豐富的 Signal 機制，讓你在任務生命週期的關鍵點插入監控邏輯：

```python
# signals.py
from celery.signals import (
    task_prerun, task_postrun, 
    task_failure, task_revoked
)
import structlog
import time

logger = structlog.get_logger()
task_start_times = {}

@task_prerun.connect
def task_started_handler(task_id, task, *args, **kwargs):
    task_start_times[task_id] = time.time()
    logger.info("task_started", 
                task_id=task_id, 
                task_name=task.name)

@task_postrun.connect  
def task_finished_handler(task_id, task, retval, state, *args, **kwargs):
    duration = time.time() - task_start_times.pop(task_id, time.time())
    logger.info("task_completed",
                task_id=task_id,
                task_name=task.name,
                state=state,
                duration_seconds=round(duration, 3))

@task_failure.connect
def task_failed_handler(task_id, exception, traceback, *args, **kwargs):
    logger.error("task_failed",
                 task_id=task_id,
                 exception=str(exception),
                 traceback=str(traceback))
    # 整合告警：發送到 Slack、PagerDuty 等
    send_alert(f"Task {task_id} failed: {exception}")
```

## Prometheus 指標：讓監控視覺化

結合 Prometheus 可以建立儀表板，即時觀察任務系統健康度：

```python
# metrics.py
from prometheus_client import Counter, Histogram, Gauge

TASK_COUNTER = Counter(
    'celery_tasks_total',
    'Total Celery tasks',
    ['task_name', 'state']
)

TASK_DURATION = Histogram(
    'celery_task_duration_seconds',
    'Task execution time',
    ['task_name'],
    buckets=[.1, .5, 1, 5, 10, 30, 60, 120, 300]
)

QUEUE_LENGTH = Gauge(
    'celery_queue_length',
    'Number of tasks in queue',
    ['queue_name']
)
```

搭配 Grafana 設定告警規則，當佇列堆積超過閾值時主動通知：

```yaml
# alerting_rules.yml
groups:
  - name: celery_alerts
    rules:
      - alert: CeleryQueueBacklog
        expr: celery_queue_length > 1000
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Celery queue backlog detected"
```

## 結論與行動建議

建立可觀測的非同步任務系統並不複雜，但需要從一開始就正確配置。建議的行動步驟：

1. **立即執行**：啟用 `result_backend` 和 `task_track_started`
2. **本週完成**：實作 Signal handlers 串接現有日誌系統
3. **本月目標**：整合 Prometheus + Grafana 建立任務監控儀表板
4. **持續優化**：根據 TASK_DURATION 數據調整 `time_limit` 參數

記住，**沒有監控的非同步任務就像是射出去不知道落點的箭**。花一點時間建立可觀測性，能為你省下無數個深夜 Debug 的痛苦。
