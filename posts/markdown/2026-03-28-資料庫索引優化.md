# 複合索引欄位順序的致命陷阱：為什麼你的查詢還是在全表掃描？

## 前言

你是否遇過這樣的情況：明明已經建立了複合索引（Composite Index），`EXPLAIN` 卻顯示查詢仍然在做全表掃描？許多中階工程師在設計索引時，往往只關注「要索引哪些欄位」，卻忽略了一個關鍵因素——**欄位順序**。

複合索引的欄位排列順序，直接決定了索引能否被有效利用。這不僅影響查詢效能，更可能讓你精心設計的索引形同虛設。本文將深入探討複合索引的「最左前綴原則」，並透過實際案例說明如何正確設計欄位順序。

## 理解最左前綴原則（Leftmost Prefix Rule）

複合索引的核心運作機制是「最左前綴原則」。當你建立一個複合索引 `(A, B, C)` 時，資料庫實際上會按照 A → B → C 的順序建立 B-Tree 結構。這意味著：

- 查詢條件包含 `A` → ✅ 可使用索引
- 查詢條件包含 `A, B` → ✅ 可使用索引
- 查詢條件包含 `A, B, C` → ✅ 可使用索引
- 查詢條件只有 `B` 或 `C` → ❌ 無法使用索引

以 MySQL 為例，假設我們有以下索引：

```sql
CREATE INDEX idx_user_search ON users (country, city, created_at);
```

以下查詢可以有效利用索引：

```sql
-- 使用完整索引
SELECT * FROM users WHERE country = 'TW' AND city = 'Taipei' AND created_at > '2026-01-01';

-- 使用部分索引 (country, city)
SELECT * FROM users WHERE country = 'TW' AND city = 'Taipei';

-- 使用部分索引 (country)
SELECT * FROM users WHERE country = 'TW';
```

但以下查詢將無法利用該索引：

```sql
-- 缺少最左欄位 country，索引失效
SELECT * FROM users WHERE city = 'Taipei';

-- 跳過中間欄位 city，只能用到 country
SELECT * FROM users WHERE country = 'TW' AND created_at > '2026-01-01';
```

## 使用 EXPLAIN 診斷索引使用狀況

在優化之前，你必須先確認索引是否真的被使用。透過 `EXPLAIN` 指令可以快速診斷：

```sql
EXPLAIN SELECT * FROM users WHERE city = 'Taipei' AND created_at > '2026-01-01';
```

重點觀察欄位：

```
+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+
| id | select_type | table | type | possible_keys | key  | key_len | ref  | rows   | Extra       |
+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+
|  1 | SIMPLE      | users | ALL  | NULL          | NULL | NULL    | NULL | 500000 | Using where |
+----+-------------+-------+------+---------------+------+---------+------+--------+-------------+
```

當 `type` 顯示 `ALL` 且 `key` 為 `NULL`，代表正在進行全表掃描。若索引正常運作，`type` 應顯示 `ref`、`range` 或 `index`。

## 欄位順序設計的三個實用原則

### 原則一：高選擇性欄位優先

選擇性（Selectivity）指的是欄位中不重複值的比例。高選擇性的欄位放在前面，能更快縮小搜尋範圍：

```sql
-- 假設 user_id 有 100 萬個不同值，status 只有 5 種
-- 較佳設計
CREATE INDEX idx_order ON orders (user_id, status);

-- 較差設計
CREATE INDEX idx_order ON orders (status, user_id);
```

### 原則二：等值查詢欄位優先於範圍查詢

範圍查詢（`>`、`<`、`BETWEEN`）會中斷索引的後續欄位使用：

```sql
-- created_at 使用範圍查詢，status 將無法利用索引
CREATE INDEX idx_log ON logs (created_at, status);  -- 較差

-- 等值查詢放前面，範圍查詢放後面
CREATE INDEX idx_log ON logs (status, created_at);  -- 較佳
```

### 原則三：考慮排序需求

如果查詢經常搭配 `ORDER BY`，將排序欄位納入索引可避免額外的檔案排序：

```sql
-- 同時滿足 WHERE 和 ORDER BY
CREATE INDEX idx_product ON products (category_id, price);

SELECT * FROM products WHERE category_id = 5 ORDER BY price LIMIT 20;
```

## 實戰案例：訂單查詢優化

假設有一張百萬筆記錄的訂單表，常見查詢如下：

```sql
SELECT * FROM orders 
WHERE merchant_id = 123 
  AND status = 'completed' 
  AND created_at BETWEEN '2026-03-01' AND '2026-03-28'
ORDER BY created_at DESC
LIMIT 50;
```

根據前述原則，最佳索引設計為：

```sql
CREATE INDEX idx_orders_search ON orders (merchant_id, status, created_at);
```

優化後使用 `EXPLAIN` 驗證：

```sql
EXPLAIN SELECT ...;
-- type: range, key: idx_orders_search, rows: 1250
```

掃描行數從原本的 100 萬降至 1,250 行，效能提升顯著。

## 結論與行動建議

複合索引的欄位順序不是隨意決定的，而是需要根據查詢模式精心設計。錯誤的順序會讓索引完全失效，正確的順序則能帶來數百倍的效能提升。

**立即行動清單：**

1. 盤點專案中現有的複合索引，使用 `EXPLAIN` 確認是否被有效利用
2. 檢視慢查詢日誌，找出可能因索引順序問題導致的效能瓶頸
3. 遵循「高選擇性優先、等值優先、考慮排序」三原則重新設計索引
4. 在測試環境驗證後再部署至正式環境

記住：索引不是越多越好，而是要「建對」比「建多」更重要。
