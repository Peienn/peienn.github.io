# 當 Composite Index 順序搞錯，查詢直接慢 100 倍：複合索引欄位順序的實戰優化指南

## 前言

你是否遇過這種情況：明明已經建立了索引，`EXPLAIN` 一看卻顯示 Full Table Scan？或者索引有用到，但效能就是不如預期？在眾多索引優化的細節中，**複合索引（Composite Index）的欄位順序**是最容易被忽略、卻影響最劇烈的關鍵因素。

許多工程師在建立複合索引時，習慣性地把欄位「照表結構順序」或「照 Query 中出現的順序」排列，卻不知道這個決定可能讓查詢效能相差數十甚至上百倍。本文將以 MySQL/PostgreSQL 為例，帶你理解複合索引欄位順序的核心原則，並提供實際診斷與優化的方法。

## 理解 Leftmost Prefix 原則

複合索引的運作遵循 **Leftmost Prefix（最左前綴）原則**。簡單來說，資料庫只能從索引的最左邊欄位開始「連續」使用索引。

假設我們有以下索引：

```sql
CREATE INDEX idx_user_status_created ON orders (user_id, status, created_at);
```

這個索引可以被以下查詢有效利用：

```sql
-- ✅ 使用 user_id（第一欄）
SELECT * FROM orders WHERE user_id = 123;

-- ✅ 使用 user_id + status（前兩欄）
SELECT * FROM orders WHERE user_id = 123 AND status = 'pending';

-- ✅ 使用全部三欄
SELECT * FROM orders WHERE user_id = 123 AND status = 'pending' AND created_at > '2026-01-01';
```

但以下查詢則**無法有效利用**此索引：

```sql
-- ❌ 跳過 user_id，直接查 status
SELECT * FROM orders WHERE status = 'pending';

-- ❌ 跳過 status，查 user_id + created_at
SELECT * FROM orders WHERE user_id = 123 AND created_at > '2026-01-01';
```

第二個例子特別容易踩雷：雖然 `user_id` 可以用到索引，但 `created_at` 的範圍查詢會退化成在已篩選的結果中逐筆掃描。

## 欄位順序的黃金法則

在決定複合索引欄位順序時，請依照以下優先順序排列：

1. **等值查詢（=）的欄位放前面**
2. **選擇性高（Cardinality 高）的欄位優先**
3. **範圍查詢（>、<、BETWEEN）的欄位放最後**

以實際案例說明，假設我們的訂單查詢模式如下：

```sql
SELECT * FROM orders 
WHERE tenant_id = 'T001' 
  AND status = 'shipped' 
  AND created_at BETWEEN '2026-01-01' AND '2026-03-31';
```

正確的索引設計應該是：

```sql
-- ✅ 推薦順序：等值欄位在前，範圍欄位在後
CREATE INDEX idx_orders_optimal ON orders (tenant_id, status, created_at);
```

而不是：

```sql
-- ❌ 錯誤順序：範圍欄位在中間，會中斷後續欄位的索引使用
CREATE INDEX idx_orders_suboptimal ON orders (tenant_id, created_at, status);
```

## 實戰診斷：用 EXPLAIN 驗證索引效率

建立索引後，一定要用 `EXPLAIN` 驗證是否如預期運作。以下是 MySQL 的診斷指令：

```sql
EXPLAIN SELECT * FROM orders 
WHERE tenant_id = 'T001' AND status = 'shipped' AND created_at > '2026-01-01';
```

重點觀察欄位：

```
+----+-------------+--------+------+-------------------+-------------------+---------+-------------+------+-------------+
| id | select_type | table  | type | possible_keys     | key               | key_len | ref         | rows | Extra       |
+----+-------------+--------+------+-------------------+-------------------+---------+-------------+------+-------------+
|  1 | SIMPLE      | orders | range| idx_orders_optimal| idx_orders_optimal| 108     | const,const |  523 | Using where |
+----+-------------+--------+------+-------------------+-------------------+---------+-------------+------+-------------+
```

- **type**：`range` 或 `ref` 代表有效使用索引，`ALL` 代表全表掃描
- **key_len**：數值越大代表越多欄位被索引使用
- **rows**：預估掃描行數，越小越好

對於 PostgreSQL，可以使用：

```sql
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders 
WHERE tenant_id = 'T001' AND status = 'shipped' AND created_at > '2026-01-01';
```

## 常見陷阱與優化建議

### 陷阱一：在高 Cardinality 欄位上做範圍查詢卻放前面

```sql
-- 假設 created_at 是高選擇性欄位，但它是範圍查詢
-- ❌ 這樣會導致 status 欄位無法被索引過濾
CREATE INDEX idx_bad ON orders (created_at, status);
```

### 陷阱二：索引欄位過多導致維護成本過高

每個索引都會增加寫入負擔。建議使用以下指令定期檢查索引使用率：

```sql
-- MySQL：檢查索引使用統計
SELECT * FROM sys.schema_index_statistics WHERE table_schema = 'your_db';

-- PostgreSQL：檢查未使用的索引
SELECT indexrelname, idx_scan FROM pg_stat_user_indexes WHERE idx_scan = 0;
```

## 結論與行動建議

複合索引欄位順序看似小細節，實則直接影響查詢效能的數量級。掌握 Leftmost Prefix 原則、遵循「等值優先、範圍在後」的排列法則，並善用 `EXPLAIN` 持續驗證，就能避免大多數索引失效的問題。

**立即行動清單：**

1. 盤點目前專案中使用的複合索引，檢查欄位順序是否符合查詢模式
2. 對慢查詢執行 `EXPLAIN`，確認索引是否被完整使用
3. 移除使用率為零的冗餘索引，減少寫入開銷
4. 在 CI/CD Pipeline 中加入慢查詢監控，及早發現索引退化問題

索引優化是一門持續迭代的功夫，從今天開始重新審視你的複合索引設計吧。
