# 何謂 SLI / SLO / SLA / Error Budget


前一章提到，SRE 主要就是透過軟體工程方法去管理系統和服務，確保服務可靠性、擴充性和效能。

假設今天主管問你：我們的產品/服務 整體穩不穩？

你怎麼回答？「還不錯」「這週都沒當機」，這些說法沒辦法量化，也沒辦法讓人信服。

而 SRE 的任務就是將這些「形容詞」量化，根據這些量化後的數字進行決策。例如：服務可用性99% or API latency < 200ms 。

## SLI (量測什麼)

SLI (Service Level Indicator)： 服務水平指標，衡量系統品質的那個數字。

- 可用性（Availability）：過去 30 天有幾 % 的時間服務是正常的？
- 延遲（Latency）：95% 的 request 在幾毫秒內回應？
- 錯誤率（Error Rate）：每 1000 個請求裡有幾個回傳 5xx？

SLI 是原始的觀測值，它本身不告訴你好不好，只告訴你現在是多少。

所以下一個我們要知道的是，要達到什麼樣的目標值。

## SLO （目標是什麼）

SLO (Service Level Objective)： 服務水平目標，定義 SLI 的目標值。

舉例來說： 「我們的服務可用性 SLO 是 99.9%」

這句話的意思就是每個月該服務可正常運作的時間要達到99.9%，換算下來，一個月只允許 43 分鐘的 downtime。

這個值怎麼訂出來的？ 服務可用性要 99.9% 還是 97.5%? 主要可以依據幾種方法去訂出
- 使用者的反應：當服務的回應時間上升到什麼情況會有人開發抱怨？
- 從歷史 SLI 校準：獲取過往歷史的 SLI 的實際數據，並把把 SLO 設定在更嚴一點的地方
- 依賴服務往回推：服務可用性不可能高於你依賴的服務，例如你的架構是 API--> EC2 --> RDS，SLP的上限就是 API * EC2 * RDS
  
SLO 值訂的越高，投入的工程/成本是以指數增加，你要訂成 99.999% 沒有問題，但是需要考量的是團隊有沒有能力去維持這個數字。另外，在 SLO 中的一個重要觀念是：**不要設 100%** ， 100％的服務可用性除了不現實之外，就算真的達到了，也會導致團隊為了維持這個數字而不敢開發新功能、升級系統。


當你訂好了 SLO ，也會定期去檢測這些 SLI 是否達標，最終目的是什麼？

-> 當然是要承諾給買你服務的客戶，你有辦法達到這些指標，人家才會來買。

---

舉例 ：今天客戶要租用一個購物車的平台服務， 
- Ａ公司有 SRE 團隊可以在合約上白紙黑色寫著 : **`服務可用性：99% + API Latency < 100ms + Error Rate < 3% ...`**
- B 公司沒有 SRE 團隊，只能告訴客戶 **`我們的服務絕對沒問題，用過都說讚`**

請問客戶會選哪個？


## SLA (協議承諾)

SLA (Service Level Agreement)： 服務水平協議，對客戶承諾的服務水平。

如果你的服務沒達到 SLA 的標準，通常會有具體的賠償條款，例如退費、服務點數之類的。
所以 SLA 一定會比 SLO 保守，中間要留緩衝。你不能對客戶承諾 99.9%，但自己的 SLO 也設在 99.9%——萬一出了什麼事你完全沒有空間處理，就直接違約了。

一個常見的設法是：SLA 設在 99.5%，SLO 設在 99.9%，SLO 超標時你就有時間在違反 SLA 之前先處理好。


## Error Budget (有多少犯錯空間)

公司開發的服務/系統賣給

如果你的 SLO 是 99.9% 可用性，那就代表你一個月有 0.1% 的時間可以是「不可用的」，換算下來大約是 43 分鐘。這 43 分鐘就是你的 Error Budget。

Error Budget 的核心是：**`它不只是用來記錄失敗的，它是用來做決策的`** 。

- Error Budget 還很多 → 表示系統夠穩，工程師可以大膽部署新功能、做架構調整
- Error Budget 快燒完 → 表示系統最近出了不少問題，這時候應該優先做穩定性工作，不要再加速推新功能

這可以幫助我們在針對自家產品，當前是要求 **`穩定`**  還是 **`系統功能開發`** 的決策中，變成了一個依據數據所做的決策，而不是靠感覺。


## 彼此間的關係

- SLI 是你量的數字，SLO 是你對自己的目標，SLA 是你對客戶的承諾。
  
- SLO > SLA , 有反應的空間和時間
  
- Error Budget & SLO &c SLA (下圖)
```
100% ──────────────────────────

         ↕  Error Budget (1%)   ← 判斷產品的推進或穩定

 99% ──────────────────────────  SLO（內部目標）

         ↕  緩衝區 (2%)          ← SLO 失守但還沒違約

 97% ──────────────────────────  SLA（對客戶的承諾）

         ↕  違約區               ← 跌破就要賠償

  0% ──────────────────────────

```

當全新專案完成並準備開始訂目標時，可以先思考簡單幾個問題，以此來決定要測量哪些SLI

1. 這個服務對使用者最重要的事是什麼？
2. 壞到什麼程度使用者會有感？
3. 我現在有能力維持到哪裡？


之後就可以埋 metrics 開始搜集數據  (1~2週)

並根據這段時間的數據，訂出第一版 SLO ( 數據 * 0.95 )，之後便時刻監控並依據實際情況調整。


### SLI

| 服務層 | SLI | SLO | 說明 |
|--------|-----|-----|------|
| Nginx | 整體可用性 | > 99.9% | blackbox_exporter 偵測 |
| Nginx | p95 回應時間 | < 300ms | 含 load balance 轉發時間 |
| API（後端） | HTTP 成功率 | > 99.5% | 5xx 錯誤率 < 0.5% |
| API（後端） | p95 回應時間 | < 200ms | 95% 的 request 在 200ms 內 |
| Redis | Cache hit rate | > 80% | 低於此值代表快取策略有問題 |
| Redis | MQ 積壓量 | < 1000 msgs | 超過代表 consumer 跟不上 |
| PostgreSQL | 查詢成功率 | > 99.5% | 連線或查詢失敗率 |
| Container | 重啟次數 | 0 次/天 | 任何 container crash 都要告警 |


---
為什麼用 p95 而不是平均值？
> 因為均值會被少數極端值稀釋，p95 代表「100 個請求裡最慢的那 5 個不算，第 95 個的回應時間」，更能反映大多數用戶的實際感受。
