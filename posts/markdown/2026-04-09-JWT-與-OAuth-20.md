# 別再把 JWT 當 Session 用：OAuth 2.0 中 Access Token 的正確存活時間設計

## 前言

在導入 OAuth 2.0 的過程中，許多團隊會直接將 Access Token（通常是 JWT 格式）的過期時間設定為數小時甚至數天，彷彿它就是傳統的 Session Token。這個看似無害的決定，實際上正在讓你的系統暴露於 Token 洩漏後的長時間風險窗口中。

身為 DevOps 工程師，我們不只要讓服務能跑，更要確保它安全地跑。本文將深入探討 OAuth 2.0 架構中 Access Token 的生命週期設計，說明為何短命 Token 搭配 Refresh Token 才是正確的實踐方式，並提供具體的實作建議。

## JWT 作為 Access Token 的致命特性：無法撤銷

JWT 是一種自包含（self-contained）的 Token 格式，驗證方只需要公鑰即可驗證其有效性，不需要查詢資料庫或呼叫認證伺服器。這是它的優點，同時也是最大的風險來源。

當一個 JWT Access Token 被竊取時，你幾乎無法立即使其失效。即使你在資料庫中標記該 Token 為撤銷狀態，分散式的 Resource Server 仍然會因為簽章驗證通過而接受它。

```bash
# 解碼一個 JWT 來查看其內容（不驗證簽章）
echo "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9..." | \
  cut -d'.' -f2 | \
  base64 -d 2>/dev/null | \
  jq .

# 輸出範例：
# {
#   "sub": "user-123",
#   "exp": 1744444800,  # 過期時間
#   "scope": "read write admin"
# }
```

這就是為什麼 Access Token 的 `exp` 值如此關鍵——它是你唯一可靠的自動失效機制。

## 正確的 Token 生命週期設計模式

OAuth 2.0 最佳實踐建議採用「短命 Access Token + 長效 Refresh Token」的雙 Token 架構：

| Token 類型 | 建議存活時間 | 儲存位置 | 用途 |
|-----------|-------------|---------|------|
| Access Token | 5-15 分鐘 | Memory / 短期儲存 | API 請求授權 |
| Refresh Token | 7-30 天 | 安全的 HttpOnly Cookie 或後端 | 換發新 Access Token |

這個設計的精髓在於：Access Token 洩漏的風險窗口被限縮到分鐘級別，而 Refresh Token 只會與 Authorization Server 通訊，攻擊面較小且可實作即時撤銷。

在 Authorization Server（以 Keycloak 為例）的設定：

```bash
# 使用 Keycloak Admin CLI 設定 Realm 的 Token 生命週期
/opt/keycloak/bin/kcadm.sh update realms/production \
  -s "accessTokenLifespan=300" \
  -s "ssoSessionIdleTimeout=1800" \
  -s "refreshTokenMaxReuse=0"

# accessTokenLifespan=300 表示 5 分鐘
# refreshTokenMaxReuse=0 表示 Refresh Token 只能使用一次（Rotation）
```

## 實作 Refresh Token Rotation 防止重放攻擊

單純縮短 Access Token 壽命還不夠，我們需要確保 Refresh Token 本身也有保護機制。Refresh Token Rotation 是一個關鍵的安全實踐：每次使用 Refresh Token 換發新的 Access Token 時，同時也換發新的 Refresh Token，舊的立即失效。

```python
# 後端處理 Token Refresh 的範例邏輯
import httpx
from datetime import datetime, timedelta

async def refresh_access_token(refresh_token: str) -> dict:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "https://auth.example.com/oauth/token",
            data={
                "grant_type": "refresh_token",
                "refresh_token": refresh_token,
                "client_id": "your-client-id",
                "client_secret": "your-client-secret",
            }
        )
    
    tokens = response.json()
    # 回傳包含新的 access_token 和 refresh_token（已輪換）
    # {
    #   "access_token": "new-jwt...",
    #   "refresh_token": "new-refresh-token...",  # 舊的已失效
    #   "expires_in": 300
    # }
    return tokens
```

當攻擊者嘗試使用已被輪換掉的舊 Refresh Token 時，Authorization Server 應該立即撤銷該使用者的所有 Token，這是偵測 Token 洩漏的重要訊號。

## 部署階段的監控與告警設計

從 DevOps 角度，我們需要建立對異常 Token 使用的可觀測性：

```yaml
# Prometheus AlertManager 規則範例
groups:
  - name: oauth_security
    rules:
      - alert: RefreshTokenReuseDetected
        expr: |
          sum(rate(oauth_refresh_token_reuse_total[5m])) > 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "偵測到 Refresh Token 重複使用嘗試"
          description: "可能存在 Token 洩漏，已觸發自動撤銷機制"
      
      - alert: AbnormalTokenRefreshRate
        expr: |
          sum(rate(oauth_token_refresh_total[5m])) 
          / sum(rate(oauth_token_refresh_total[1h] offset 1d)) > 3
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Token Refresh 頻率異常升高"
```

## 結論與行動建議

將 Access Token 當作短命的「入場券」而非長期的「身分證」，是 OAuth 2.0 安全實踐的核心觀念。作為 DevOps 工程師，你現在應該：

1. **審視現有設定**：檢查你的 Authorization Server，將 Access Token 壽命調整至 15 分鐘以內
2. **啟用 Refresh Token Rotation**：確保每次 Refresh 都會輪換 Token
3. **建立監控機制**：對 Token 重複使用、異常 Refresh 頻率建立告警
4. **與前端團隊協作**：確保 Client 端有正確實作靜默刷新（Silent Refresh）機制

安全性往往在那些「反正能跑就好」的決定中被犧牲。花一個下午重新審視你的 Token 策略，可能省下未來無數
