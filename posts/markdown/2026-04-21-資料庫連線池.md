# 資料庫連線池耗盡：從告警到根因分析的實戰排查指南

## 前言

凌晨三點，手機震動，PagerDuty 告警：「Database connection pool exhausted」。這個場景對於中階 DevOps 工程師來說絕不陌生。連線池耗盡往往不是單純的「調大 pool size」就能解決，背後可能隱藏著慢查詢、連線洩漏、或是應用程式層的設計缺陷。

本文將帶你從監控告警出發，建立一套系統化的排查流程，快速定位連線池耗盡的真正根因，而非只是治標不治本地重啟服務。

## 第一步：確認現場狀態與基本指標

當收到連線池告警時，第一件事是確認資料庫端的實際連線狀態。以 PostgreSQL 為例，立即執行以下查詢：

```sql
-- 查看當前連線數與狀態分布
SELECT state, count(*) 
FROM pg_stat_activity 
GROUP BY state;

-- 查看各應用程式的連線佔用
SELECT application_name, client_addr, count(*) 
FROM pg_stat_activity 
GROUP BY application_name, client_addr 
ORDER BY count DESC;
```

同時檢查應用程式端的連線池指標。若使用 HikariCP（Java 生態系最常見的連線池），可透過 JMX 或 Prometheus metrics 取得關鍵數據：

```bash
# 透過 Prometheus 查詢 HikariCP 指標
hikaricp_connections_active{pool="master"}
hikaricp_connections_pending{pool="master"}
hikaricp_connections_timeout_total{pool="master"}
```

重點觀察 `active connections` 是否持續維持在 `maximum pool size` 附近，以及 `pending threads` 是否持續堆積。

## 第二步：識別連線洩漏的特徵

連線洩漏是最常見卻最難察覺的根因。典型特徵是：連線數緩慢上升，直到觸頂後服務開始報錯，重啟後暫時恢復，但問題會週期性重現。

檢查是否有長時間處於 `idle` 狀態卻未被釋放的連線：

```sql
-- 找出閒置超過 10 分鐘的連線
SELECT pid, application_name, state, 
       now() - state_change as idle_duration,
       query
FROM pg_stat_activity
WHERE state = 'idle'
  AND now() - state_change > interval '10 minutes';
```

在應用程式層，設定連線池的洩漏偵測機制。以 HikariCP 為例：

```yaml
# application.yml
spring:
  datasource:
    hikari:
      leak-detection-threshold: 30000  # 30 秒未歸還視為洩漏
      connection-timeout: 10000
      maximum-pool-size: 20
```

啟用後，當連線借出超過閾值，日誌會印出完整的 stack trace，直接定位到哪段程式碼忘記關閉連線。

## 第三步：追蹤慢查詢與長事務

另一個常見根因是慢查詢或長事務佔住連線。即使程式碼正確歸還連線，若單一查詢執行過久，連線池的可用連線會被逐漸消耗。

```sql
-- 找出執行中的慢查詢（超過 30 秒）
SELECT pid, now() - query_start as duration, 
       state, query
FROM pg_stat_activity
WHERE state = 'active'
  AND now() - query_start > interval '30 seconds'
ORDER BY duration DESC;

-- 找出長時間未結束的交易
SELECT pid, now() - xact_start as transaction_duration,
       query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
  AND now() - xact_start > interval '5 minutes';
```

若確認是特定查詢造成問題，可搭配 `pg_stat_statements` 擴充套件進行歷史分析：

```sql
SELECT query, calls, mean_exec_time, max_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;
```

## 第四步：建立預防性監控與告警

排查完畢後，務必建立完整的監控機制避免問題復發。以下是建議的 Prometheus alerting rules：

```yaml
groups:
  - name: database_pool_alerts
    rules:
      - alert: ConnectionPoolNearExhaustion
        expr: |
          hikaricp_connections_active / hikaricp_connections_max > 0.8
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "連線池使用率超過 80%"
          
      - alert: ConnectionPoolPendingHigh
        expr: hikaricp_connections_pending > 5
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "有超過 5 個執行緒等待取得連線"
```

## 結論與行動建議

連線池耗盡的排查需要同時從資料庫端與應用程式端雙向夾擊。建議你在下次維護時段執行以下行動：

1. **啟用洩漏偵測**：設定 `leak-detection-threshold`，讓問題提早曝光
2. **部署完整監控**：確保 active、pending、timeout 三項指標皆有告警
3. **定期審視慢查詢**：將 `pg_stat_statements` 報表納入週報流程
4. **合理化 pool size**：pool size 不是越大越好，通常設為 `(CPU 核心數 * 2) + 磁碟數` 即可

記住，調大連線池只是止痛藥，找到根因才是真正的解方。
