# MongoDB Replica Set 故障轉移：從原理到實戰監控指南

## 前言

在生產環境中運行 MongoDB，高可用性是不可妥協的需求。Replica Set 作為 MongoDB 內建的高可用方案，能在主節點故障時自動完成故障轉移（Failover）。然而，許多團隊在部署完 Replica Set 後就「設定完忘記」，直到真正發生故障時才發現選舉失敗、應用程式連線中斷、資料不一致等問題。

本文將深入探討 Replica Set 故障轉移的運作機制，並提供實用的監控策略與工具配置，幫助你在問題發生前就做好準備。

## Replica Set 選舉機制解析

MongoDB Replica Set 採用 Raft-like 共識演算法進行領導者選舉。當 Secondary 節點偵測到 Primary 無回應（預設 10 秒心跳超時），就會發起選舉。

理解選舉的關鍵參數：

```javascript
// 查看當前 Replica Set 配置
rs.conf()

// 重要參數說明
{
  "settings": {
    "electionTimeoutMillis": 10000,  // 選舉超時時間
    "heartbeatTimeoutSecs": 10,       // 心跳超時
    "catchUpTimeoutMillis": -1        // 新 Primary 追趕時間
  }
}
```

選舉成功需要**多數派**（majority）同意。這意味著 3 節點叢集最多容忍 1 節點故障，5 節點叢集容忍 2 節點故障。常見的錯誤是部署偶數節點（如 2 或 4 個），這會大幅降低容錯能力。

## 故障轉移期間的應用程式行為

故障轉移通常在 10-30 秒內完成，但這段期間應用程式的行為取決於連線字串配置：

```bash
# 正確的連線字串格式（包含所有節點）
mongodb://mongo1:27017,mongo2:27017,mongo3:27017/mydb?replicaSet=rs0&readPreference=secondaryPreferred&w=majority&retryWrites=true

# 錯誤示範：只連單一節點
mongodb://mongo1:27017/mydb  # 故障轉移後會完全斷線
```

關鍵參數配置建議：

- `retryWrites=true`：自動重試因故障轉移中斷的寫入操作
- `w=majority`：確保寫入被多數節點確認，避免 rollback
- `readPreference=secondaryPreferred`：讀取可分散到 Secondary，降低 Primary 壓力

## 實戰監控指標與告警設定

被動等待故障發生是最糟的策略。以下是使用 Prometheus + MongoDB Exporter 的監控配置：

```yaml
# prometheus-mongodb-rules.yml
groups:
  - name: mongodb_replicaset
    rules:
      # 偵測 Replica Set 成員數量異常
      - alert: MongoDBReplicaSetMemberDown
        expr: mongodb_rs_members_health == 0
        for: 30s
        labels:
          severity: critical
        annotations:
          summary: "MongoDB RS member {{ $labels.member_idx }} is down"

      # 偵測複寫延遲過高
      - alert: MongoDBReplicationLag
        expr: mongodb_rs_members_optimeDate{member_state="SECONDARY"} - mongodb_rs_members_optimeDate{member_state="PRIMARY"} > 30
        for: 1m
        labels:
          severity: warning
        annotations:
          summary: "Replication lag exceeds 30 seconds"
```

在 MongoDB Shell 中快速檢查健康狀態：

```javascript
// 檢查 Replica Set 狀態
rs.status().members.forEach(m => {
  print(`${m.name}: ${m.stateStr}, health: ${m.health}, lag: ${m.optimeDate}`)
})

// 檢查 oplog 大小（攸關可容忍的複寫中斷時間）
rs.printReplicationInfo()
```

## 故障演練：主動驗證容錯能力

定期進行故障演練是確保高可用的唯一方法。以下是安全的演練步驟：

```javascript
// 步驟 1：確認當前 Primary
rs.isMaster().primary

// 步驟 2：強制當前 Primary 降級（會觸發選舉）
rs.stepDown(60)  // 60 秒內不會重新成為 Primary

// 步驟 3：觀察選舉過程
rs.status()
```

演練檢核清單：
1. 記錄故障轉移完成時間（應小於 30 秒）
2. 確認應用程式自動重連成功
3. 驗證沒有資料遺失或 rollback
4. 確認監控告警正確觸發

## 結論與行動建議

Replica Set 故障轉移看似自動化，但「自動」不等於「可靠」。以下是你應該立即執行的行動：

1. **檢查連線字串**：確保包含所有節點並啟用 `retryWrites`
2. **部署監控**：至少監控成員健康狀態與複寫延遲
3. **每季演練**：安排非尖峰時段進行 `rs.stepDown()` 測試
4. **審視節點數量**：確保是奇數節點，生產環境建議至少 3 節點跨可用區部署

高可用不是部署完成的那一刻達成的，而是透過持續監控與演練來維持的。
